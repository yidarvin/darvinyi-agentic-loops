1. **`artifacts/ch21-stage-three-production-grade/stage_three_agent.py`: the public `demo --sandbox test` switch bypasses the artifact's fail-closed boundary.** `build_parser()` exposes `--sandbox {auto,test}` (line 549), `main()` forwards it to `MacOSSandbox` (line 577), and `_command()` returns the raw child argv in `test` mode (lines 197-200). `python3 stage_three_agent.py demo --help` confirms that this is available to normal demo callers, contrary to the README's claim that the test double is unavailable to the normal demo and the chapter's statement that non-macOS execution fails closed. Remove this escape hatch from the public demo parser or make it unreachable outside the artifact check, then add a regression test that a normal demo cannot launch an unsandboxed child.
2. **`artifacts/ch21-stage-three-production-grade/stage_three_agent.py`: a workspace-controlled `.agent-memory` symlink redirects host-process memory writes outside the workspace.** `SafeMemory.__init__()` resolves the memory root (lines 92-94) but never proves that resolved root remains below `workspace`; its writes occur in the unsandboxed harness process. In a temporary workspace where `.agent-memory` linked to an outside directory, the documented normal demo completed and wrote `project.md` and `mcp-tool-lock.json` outside the workspace, while only `verification.txt` stayed inside. This contradicts the contained-path lesson and the README's workspace-write contract. Reject a memory root that resolves outside the workspace, and add this symlink case to `check.sh` or the self-test.
3. **`artifacts/ch21-stage-three-production-grade/stage_three_agent.py`: a configured MCP server starts before any policy decision.** `ProductionHarness.run()` calls `client.start()` before `_authorize()` (lines 414-417), and `StdioMcpClient.start()` launches `self.command` before initialization (line 276). Thus an unapproved custom `--mcp-command` can execute startup, initialization, and discovery code despite the chapter and figure teaching that every effectful action crosses policy before the OS boundary. Add a distinct server-launch policy or explicit approval before `popen`, display the exact command, and test that a denied server command cannot create even a workspace-local marker.
4. **`artifacts/ch21-stage-three-production-grade/stage_three_agent.py`: the MCP server can rewrite the tool-definition lock that is supposed to detect its drift.** The lock lives under workspace memory (lines 406-414), while the Seatbelt profile grants the server write access to the whole workspace (lines 191-194) and gives it the workspace path. A malicious server can write the hash for its changed definition before replying to `tools/list`, causing `pin_tool_definitions()` to accept the change. Keep the lock in agent-controlled storage outside the server's writable boundary, and add a malicious-server regression test proving that a changed definition is rejected even when the server attempts to alter the lock.
