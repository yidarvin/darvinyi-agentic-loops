An untrusted tool result or tool definition can become attacker-controlled content when a
host places it in a model context. That dataflow, not an MCP wire-protocol defect, is the
MCP security surface. The risk becomes an autonomous loss when the same host also grants
sensitive capability and an external or destructive action. MCP makes those integrations
easy to compose, but it does not require a particular model-context or user-interaction
architecture. Prompt injection is the root risk. Current evidence does not establish
detection as a reliable universal boundary. Trusted policy enforcement can constrain
defined flows within its threat model. This chapter is where you learn what goes wrong and
which controls materially change the outcome.

## Data becomes instructions

Simon Willison coined the term prompt injection in September 2022, by analogy to SQL
injection. The failure is the same: trusted and untrusted content share one channel. In
SQL you fix it with parameterization, because a query has a schema that separates code
from data. Natural language has no schema and no delimiter, so there is no analog to
escaping. When a host puts a tool result in the same context window as its system prompt,
the model processes both through the same next-token prediction. The model has no plane
where host instructions live and the data cannot. Everything in that context is one stream
of tokens, and the model was trained to follow instructions wherever it finds them.

This is not jailbreaking. Jailbreaking coaxes a model into embarrassing output. Injection
puts attacker-controlled instructions into a trusted execution context that holds real
tools and real credentials. The useful mental model is taint tracking. Treat any
exposure to untrusted content as a taint event. Once the agent ingests attacker-controlled
tokens, assume everything derived on that turn is compromised, and block any action that
could send data out. Willison's security framing is worth keeping in view: 99% filtering is
a failing grade when it leaves an adaptive attacker a remaining failure case to keep
probing. A one-percent failure rate does not make a particular attempt certain to succeed,
but it leaves a consequential path for repeated adversarial search.

## The lethal trifecta

Willison's sharpest framing, from June 2025, names the exact exploitable combination. Call
it the lethal trifecta: access to private data, exposure to untrusted content, and the
ability to communicate externally. Removing one leg breaks this particular autonomous
exfiltration path. It does not make the remaining system universally safe. All three in one
session is the condition under which your data can leave the building without another
boundary intervening. An MCP host can assemble those capabilities by connecting tools from
different sources, and a single tool can supply more than one leg at once. A tool that
retrieves email can be both a private-data source and an untrusted-content source, which
means an attacker can email your agent and tell it what to do.

Read the trifecta as a design constraint, not a warning. It tells you which capability
combinations to refuse before you connect a server. For a defined flow, remove or gate a
leg at the trusted host. Inspection can lower exposure, but it does not prove that an
adaptive payload cannot close the path.

## Indirect injection is the delivery machine

Direct injection is a user, or someone posing as one, typing malicious instructions.
Indirect injection is the dangerous variant for hosts that place retrieved content in model
context: the instructions arrive inside data the agent retrieves. A web page, a document, a
Jira ticket, a GitHub issue. An MCP host may use tools to fetch that content, then combine it
with private tools and an outbound action. Two chains show how little the attacker needs.

The GitHub toxic-agent flow, disclosed by Invariant Labs in May 2025, is the canonical
case. An attacker files a malicious issue in a public repository. The victim later asks
their agent to look at open issues. The agent ingests the injected instructions and uses
its own legitimate GitHub tools to pull data from a private repository and publish it in a
public pull request. The proof of concept exfiltrated private repository names, a
relocation plan, and salary data. This is not a bug in the GitHub server's code, so GitHub could not fix it with a
server-side patch. It is a property of wiring untrusted content to private tools and a
public sink. A separate July 2026 GitLost demonstration against GitHub Agentic Workflows
reported that adding the word "Additionally" bypassed restrictive guardrails in that proof
of concept. The mechanism is product-specific; the lesson is not.

EchoLeak (CVE-2025-32711, Microsoft CVSS 3.1 9.3), found by Aim Labs in Microsoft 365
Copilot, is widely described as the first documented zero-click prompt-injection
exfiltration in a production LLM system. A malicious email carries hidden instructions
framed as directions to its recipient and omits AI, assistant, and Copilot terminology,
which made XPIA detection difficult. When retrieval later pulls the email into context,
Copilot can place sensitive data and an attacker endpoint in a reference-style markdown
image URL. A direct `evil.com` image was blocked by `img-src` CSP. The successful chain
sent the automatic image request to a CSP-allowlisted Microsoft Teams `asyncgw` proxy,
which then fetched the attacker endpoint carrying the secret. Aim Labs called the pattern
an LLM Scope Violation. It chained four separate bypasses to get there, and it was patched
with no known exploitation in the wild. Its value is as proof that the attack class works
end to end against a real, well-resourced product.

## The poisoned surface: descriptions, schemas, outputs

Untrusted content is not only what a tool returns. Tool metadata becomes an injection
surface when a host supplies it to the model, and the reader who stops at "sanitize tool
results" misses most of it.

A Tool Poisoning Attack, named by Invariant Labs in April 2025, hides instructions in a
tool's description. A host may supply that description to the model so it can decide whether
and how to call the tool, as [Resources, Tools, and Prompts](/resources-tools-and-prompts)
showed, while a simplified UI may show only a name. A description can nest an `<IMPORTANT>`
block telling the agent to read `~/.ssh/id_rsa` and smuggle it into a parameter, and the
poison persists across every session that loads the server. CyberArk widened this in late
May 2025 with Full-Schema Poisoning. Parameter names, types, required arrays, defaults, and
enums become part of the attack surface when the host parses them into model context. A
scanner that reads only the description misses them. They also demonstrated Advanced Tool
Poisoning, where the injection lives in a tool's runtime output. A calculator returns a fake
error demanding a file "to proceed," the model treats it as a real requirement, reads the
file, and resends it. Static catalog or metadata inspection cannot catch that runtime payload
because it did not exist until the tool ran.

Two more shapes round out the surface. A rug pull serves a benign definition at approval
and swaps in a malicious one later. Semgrep documented that the historical specification had
no requirement to broadcast tool changes or re-approve changed capabilities. Tool shadowing appears when an agent host
aggregates descriptions from connected servers into one model context without an isolation
or dataflow boundary. A malicious server's description can then inject instructions about a
different, trusted server's tools, for example that every email must be blind-copied to the
attacker, and the trusted server complies because its own definition never changed. The host
must impose that boundary. Invariant's finding was that the malicious server need not even
call the trusted tool. Shadowing is enough. The MCPTox benchmark (AAAI) puts numbers on the
model side: across 45 live servers and 1,348 malicious cases, the average attack success
rate was 36.5%; across its 20 evaluated settings, more capable models were often more
susceptible, because better instruction-following can also follow attacker instructions.

## The confused deputy, and the token you must never pass through

The trifecta and tool poisoning are model-layer problems. The confused deputy is the auth
problem an MCP proxy can make acute. A deputy is a privileged intermediary. An MCP server
holds broad tokens and keys, and the attack tricks it into wielding that authority on the
wrong party's behalf. The MCP Security Best Practices document describes a specific proxy
failure with four preconditions: the proxy uses a static client ID at a third-party
authorization server, allows MCP clients to dynamically register, encounters a prior
third-party consent cookie, and does not require per-client consent before forwarding to
that third party. A user first authorizes the normal flow, which leaves the consent cookie.
An attacker then sends a malicious link with a crafted authorization request, an attacker
redirect URI, and a new dynamically registered client ID. When the user follows it, the
cookie can skip consent, the code redirects to the attacker, and the attacker exchanges it
for tokens. Per-client consent at the MCP proxy, before that third-party authorization flow,
is the required mitigation.

The mechanical defense is now written into the specification in the imperative. The
2025-11-25 HTTP authorization profile states that an MCP server MUST validate a presented
token is intended for its own resources, reject one that is not, and MUST NOT pass the
client's token through to a downstream API. An `aud` claim is a common JWT-style way to make
that intended recipient visible. Opaque or introspected tokens can establish it another way.
A forwarded token collapses two trust boundaries into one, defeats the intended-recipient
check, and turns a compromised server into a credential laundromat. The Security Best
Practices document details the bypassed controls, indistinguishable clients, misleading logs,
and cross-service reuse that follow. The normative machinery includes intended-recipient validation at the MCP server, RFC
8707 Resource Indicators, protected-resource metadata, OAuth 2.1 with PKCE, and no
client-token transit. When an MCP server acts as an OAuth client to an upstream API, that API
uses a separate token issued by its authorization server. Other integrations can use another
independently authorized mechanism, but none may transit the client's token. RFC 8693 Token
Exchange is one possible issuance pattern when an authorization system supports it; the MCP
authorization specification does not mandate it. These are requirements of the optional HTTP
authorization profile. A stdio transport has a different trust boundary, and the specification
says stdio implementations should not run that HTTP authorization flow. The failure this
prevents is easy to build without noticing. Consider an illustrative `/mcp/orders` endpoint and a
`/mcp/billing` endpoint behind the same identity provider. If billing validates only the
token's issuer, an orders token sails through because both endpoints trust the same signer,
and the intended-recipient boundary exists on paper only.

The blast radius is set by how broad the credential is. The GitHub flow was amplified by a
wide personal access token, so a single injection could reach any repository the token
allowed. Astrix identified 5,205 GitHub repositories, downloaded their READMEs, and used an
LLM-assisted analysis of credential signals. Its reported 88% credential use, 53% long-lived
static-secret use, and 8.5% OAuth use are directional repository-level signals, not a live
server census. Least privilege is the lever: scope the agent's credential to exactly the
resources it needs, and a hijacked agent inherits a small authority instead of the keys to
the kingdom.

## Classic bugs, wired to an autonomous agent

The other half of the surface is ordinary implementation flaws exposed when an agentic host
gives a non-deterministic caller tool access. Equixly's March 2025 assessment of selected
popular MCP implementations reported command injection in 43%, path traversal or arbitrary
file read in 22%, and SSRF in 30%. Its command-injection examples include the familiar pattern
of user parameters concatenated into a shell call. Path traversal and SSRF are separate classes
that need their own path and egress controls. These vendor results are directional, not a deployment census. Trend
Micro correlated public-directory metadata for 9,695 unique entries with its prior automated
analysis and randomized manual verification, then reported confirmed issues in 2,259 of
them. The tooling around MCP is an attack surface in its own right: Oligo reported
CVE-2025-49596, a remote code execution in MCP Inspector versions below 0.14.1 (CVSS 9.4).
The browser-side route requires its unauthenticated local Inspector proxy to already be
running, then a malicious page can send it the request. JFrog characterized CVE-2025-6514 as the first documented
full RCE on a client's own operating system from a remote server, through the widely
installed `mcp-remote` proxy (CVSS 9.6). Backslash found the single most common issue was
servers bound to `0.0.0.0`, reachable by anyone on the network. These are the threats
[Building a Real MCP Server](/building-a-real-mcp-server) fixes at the server: parameterized
execution instead of string-built shell commands, path canonicalization against an allowlisted
root, SSRF egress blocking, and binding to localhost.

## Why detection is a backstop, not a universal boundary

It is tempting to answer all of this with a model that detects malicious input. Current
evidence does not support that as a reliable universal boundary. "The Attacker Moves Second"
(arXiv 2510.09023), by researchers across OpenAI, Anthropic, and Google DeepMind, used
gradient descent, reinforcement learning, and human-guided search to bypass twelve recent
defenses under its adaptive threat models. Attack success exceeded 90% against most of those
twelve tested defenses, and the majority had originally reported near-zero success. That does
not prove every detector fails in every setting. It does show why a detector alone cannot
prove a sensitive agent flow safe. Meta's Llama Prompt Guard 2 reports 97.5% recall at a 1%
false-positive rate on its private held-out English direct-jailbreak benchmark. That is useful
evidence, not a universal injection guarantee. EchoLeak avoided a detector by never
mentioning AI, and Prompt Overflow shows how segment-scoring guards can fail when a payload
is split so no segment scores high. A classifier can lower attack rate within a tested
distribution. It should not be the only control that authorizes a sensitive action.

## Defenses that constrain defined flows

Trusted enforcement controls share one property: for a defined policy and threat model, they
change which actions are possible instead of estimating whether text is malicious.
Least-privilege scoping is the first. Docker's Gateway demonstration implements a "one
repository per session" interceptor policy that blocks the public-to-private jump the
toxic-agent flow depends on, because the injected read of a second repository is not
permitted. Intended-recipient token validation with no passthrough, and a separate credential
when the server acts as an OAuth client to an upstream API, constrain the confused-deputy flow
the same way.
Human-in-the-loop approval remains useful for irreversible, destructive, and
exfiltration-capable actions, though it is not sufficient if the approval interface trains
people to click through every request.

Two frameworks generalize the trifecta into a policy you can enforce. Meta's Agents Rule of
Two, published in October 2025, says that until injection is reliably detectable, an agent
session should satisfy no more than two of: process untrustworthy input, access sensitive
data, and change state or communicate externally. If all three are needed without starting a
new session, Meta says the agent must not operate autonomously and needs supervision, through
human approval or another reliable validation mechanism. Meta also describes a safe one-way
transition between configurations when it breaks the attack chain. This chapter takes the more
conservative deployment policy of a fresh context plus human review for a task that genuinely
needs all three. The framework remains useful, not a sufficient proof. Meta calls it a
supplement, and Noma's analysis notes that property-counting can mislead when one tool supplies
two legs at once, as in EchoLeak.
Willison's Dual LLM pattern uses a privileged model that can call tools and never sees raw
untrusted content. A quarantined model handles that content, while a controller hands the
privileged model symbolic variables instead of the raw or tainted text. CaMeL (arXiv
2503.18813) is a distinct architecture: a protective layer extracts control and data flows
from the trusted query, uses a restricted interpreter, and enforces capability policy at
tool calls. Under its interpreter, policy, and threat assumptions, it reports provable
security on 77% of AgentDojo tasks against 84% undefended, a real utility cost for a bounded
guarantee. For a sensitive flow, enforcing that tainted data cannot reach a capability is
stronger than estimating that a prompt is probably benign.
Description integrity addresses the post-approval rug-pull gap: Trail of Bits'
mcp-context-protector blocks a newly seen server pending manual review of its instructions,
descriptions, and parameter descriptions, then pins that reviewed configuration and checks
every message before it enters the context window. A changed definition is caught, but the
review process does not establish that an initial definition was safe.

<Callout kind="danger">
No current detector is a universal solution to prompt injection. Adaptive attackers achieved
more than 90% success against most of the 12 defenses tested in one study. Treat an injection
detector as defense in depth, not as the only control authorizing a sensitive flow. When a
task combines untrusted input with sensitive data, place trusted policy enforcement around
both destructive and exfiltration-capable actions. Closed source, unmaintained, `0.0.0.0`-
bound, or broad-static-token servers deserve elevated review and deployment-specific controls.
</Callout>

## Run it

The artifact is a pair of local stdio MCP servers: one deliberately vulnerable and one
hardened. Both implement the JSON-RPC lifecycle, `initialize`, `notifications/initialized`, `tools/list`,
and `tools/call`. The readable walkthrough uses the same server core; `--test` launches each
endpoint, sends those messages for every attack, then lets a credulous client follow
instructions from a tool result or changed catalog on purpose. That is the vulnerability
under study. The scoreboard asks one question: did a modeled secret cross the trust boundary?
On the hardened programmed paths, it does not. Run the walkthrough to watch each exploit and
block, run `--test` for protocol-level assertions, and run `--attack 3` to watch a valid
MCP-ingress token meet authorization policy before any downstream call.

The lab models `http_get` as an external sink that fails closed until the trusted host records
human approval. It separately sends a modeled secret straight in a `tools/call` payload, with
no preceding modeled private read, and blocks that request too. A caller cannot turn missing
source metadata into permission to exfiltrate. A production host can grant narrower,
separately authorized egress paths, but the sensitive sink must remain a trusted boundary.

The servers use a deterministic, already-validated token fixture to teach intended-recipient
validation, exact scope, and no-transit rules. The fixture uses an `aud` field to make the
JWT-style case visible. It does not implement live OAuth, JWT signature validation,
protected-resource discovery, or an authorization server. The artifact names that boundary
plainly so the reader does not mistake a safe local fixture for production authentication.

## Exercises

<ExerciseCard n="01" title="Assemble the trifecta from your own tools">
List the MCP tools you actually run in one agent. Tag each as a source of untrusted content
[A], a source of private data [B], an exfiltration sink [C], or more than one. Find a single
session that holds all three legs. That session is the one an attacker can turn against you,
and naming it is the first defensive move.
</ExerciseCard>

<ExerciseCard n="02" title="Watch the control decide the outcome">
Run `python3 mcp_security.py --attack 4`. Read the vulnerable run: the injection is in the
tool's output, not its description. Now explain in one sentence why the trusted full-catalog
integrity control that stops the reviewed-baseline rug pull in attack 2 cannot stop attack 4,
why a scanner is only a backstop, and why the egress gate can. This is the difference between
checking definitions and constraining actions.
</ExerciseCard>

<ExerciseCard n="03" title="Forward a token and break the boundary">
Run `python3 mcp_security.py --attack 3`. The hardened path accepts the client's
`aud=acme-mcp` token at MCP ingress, then denies the injected `read_billing` action before
Billing is called. Compare that with `read_orders`, which uses a distinct
`aud=billing`, `read:orders` token. In a scratch change, replace the hardened policy denial
with the vulnerable issuer-only Billing call and re-run `--test`. State the MCP `MUST NOT`
you violated, why the client token must never transit downstream, and what the attacker gains.
</ExerciseCard>

<ExerciseCard n="04" title="Apply the Rule of Two to a real design">
Take one agentic feature you want to ship that needs all three legs, for example
"summarize this customer's private tickets from an inbound support email and reply." Redesign
it to satisfy no more than two legs per session. Decide where the fresh context begins and
which single action you would gate behind a human, and write down what the agent can no
longer do as a result.
</ExerciseCard>
