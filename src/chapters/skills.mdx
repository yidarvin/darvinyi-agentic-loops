A skill is a folder that packages procedural knowledge as an on-demand capability. Its
contract is a Markdown file named `SKILL.md`: a few lines of YAML describe the capability,
then a body tells an agent how to use it. Optional scripts, reference documents, and
assets live beside it. The interesting property is not the file format. It is the loading
discipline: an agent can see enough metadata to route work without paying to carry every
workflow in its context window.

The `SKILL.md` convention originated with Anthropic's Agent Skills and became an open
standard in December 2025. The portable core is deliberately small. Individual harnesses
decide discovery, explicit invocation, permissions, and exactly how long loaded content
remains resident. This chapter uses Anthropic's documented implementation for concrete
numbers, but the design pattern is vendor-neutral: [MCP](/resources-tools-and-prompts)
connects an agent to tools and data; a skill supplies the procedure for using them.

## A folder, a file, and a description

The definition is deliberately minimal: a skill is a directory containing a `SKILL.md`
file, optionally alongside bundled scripts, reference documents, and assets. The
`SKILL.md` begins with YAML frontmatter carrying two required fields and continues as an
ordinary Markdown body.

```markdown
---
name: changelog-entry
description: Formats and validates a Keep a Changelog entry. Use when adding a
  changelog entry, recording a change, or editing CHANGELOG.md.
---
# Changelog entry

Add a single line under the Unreleased section. Pick a type, write one
imperative line, then run scripts/validate_entry.py before you write it.
```

At the portable core, `name` is the identifier and `description` is the routing contract.
The Agent Skills reference implementation restricts names to lowercase letters, digits,
and hyphens, caps them at 64 characters, and forbids certain vendor names. It caps a
description at 1024 characters and asks it to state both what the skill does and when to
reach for it. Treat those as compatibility rules for that implementation, not as the
entire meaning of a skill. A harness may also expose the name as an explicit command.
By convention `scripts/` holds executable code for deterministic work, `references/`
holds documents the agent reads only when it needs them, and `assets/` holds templates or
fonts used in the skill's output. The recurring analogy is a new-hire onboarding guide:
not a rewrite of the employee, a packet of the context and procedure the job needs. One
general agent plus a library of capability packages, rather than a different architecture
per domain.

## Progressive disclosure

The loading model is the design, and it has a name: progressive disclosure. Content
reveals itself in three levels, each loaded at a different moment and costing the context
window a different amount.

In Anthropic's implementation, level one is metadata: the `name` and `description` of
every installed skill injected into the system prompt at startup. Its documentation puts
this near a hundred tokens per skill. Independent measurements land lower, around thirty
to fifty tokens each by Firecrawl's count, so a hundred installed skills cost roughly
three to five thousand tokens at session start. Level two is the `SKILL.md` body, loaded
after the harness chooses the skill, with a target of fewer than five thousand tokens and
500 lines. Level three is everything bundled: reference docs read on demand and scripts
executed without needing to first read their source into context. Because an agent with a
filesystem does not have to read a file to know it exists, the bundle can be effectively
unbounded. Anthropic's shorthand for the discipline is that "the context window is a
public good," a principle [Context-Window Economics](/context-window-economics) makes the
master constraint of the whole book.

Set this against a large MCP server, whose tool definitions typically load in full at
connection time. GitHub's official server alone consumes tens of thousands of tokens
before the agent does anything, where a skill announces itself in fifty. The saving is
real, and it is not free. Progressive disclosure trades startup tokens for runtime
filesystem round-trips: the model has to `bash` out to read the body and any referenced
file, which one analysis from MCPJam estimated at three to four times the tool calls of
loading everything upfront. Upfront loading buys latency and immediate reasoning about
which tool to call; progressive disclosure buys a clean context window. Which one wins is
the argument of the next chapter, Skill or Server, and the honest answer there is often
a hybrid.

## The description does the discovering

Skills are often model-invoked. A harness exposes the installed metadata, then the agent
decides whether a task warrants opening a body, much as it decides which tool to call. In
that design the `description` does the discovering: when a task matches it closely enough,
the harness loads the full `SKILL.md`; when it does not, the body stays on disk. Everything
hangs on this small routing contract, which makes it the highest-leverage line in the
file.

Two rules follow directly. Write it in the third person, because it is injected into the
system prompt as fact about the world, not addressed to anyone: "Formats and validates a
changelog entry," never "I can help you format" or "You can use this to." And state both
halves, what it does and when to use it, with the concrete words a matching task would
contain. Anthropic's own PDF skill is unashamedly pushy about the second half: use this
skill whenever the user wants to do anything with PDF files, and if the user mentions a
`.pdf` file or asks to produce one, use this skill. A description can match perfectly and still not fire. The model only reaches
for a skill on tasks it cannot already handle in one step. Ask it to read a single
PDF and it just reads it. Ask for a multi-step, format-sensitive operation and the skill
reliably triggers. Discovery is a judgment about difficulty, not only about keywords.

Model invocation is not the only door. Some surfaces offer an explicit command, and some
offer controls that make a skill user-only or model-only. Claude Code's
`disable-model-invocation` and `user-invocable` frontmatter fields are useful examples.
They are surface extensions, not portable assumptions. Design the body so either route
leads to the same workflow, then test both routes where the target harness supports them.

## Code that can stay out of the window

The third level is where skills stop being clever prompts and become something a prompt
cannot be. In a harness with a filesystem and a shell, a bundled script can be executed
rather than read. When the model runs `validate_entry.py`, the harness can return its
output without first loading the script's source into context. The agent can still inspect
that source when it needs to, which is why audit remains mandatory. The reasoning is
economic and it is about correctness. Sorting a list by generating tokens is far more
expensive, and far less reliable, than running a sort. Anywhere a task is deterministic,
shipping the code beats asking the model to reconstruct it every time. Willison's
`slack-gif-creator` skill shows the pattern at its cleanest: the model runs a bundled
builder, then a bundled validator checks the result against Slack's file-size limit, and
if it fails the model adjusts and runs again. A feedback loop the skill author wrote once,
executed deterministically, correcting the model's non-deterministic output.

The caveat that bites in practice is that a portable folder does not create a portable
execution environment. Network egress, package installation, filesystem reach, and
available binaries belong to the harness and its policy. A skill that quietly assumes it
can `pip install` a dependency works in one place and fails in another. Start from the
smallest dependency set, state every requirement, and fail clearly when the environment
cannot provide one. Then test the actual surfaces you intend to support.

## The authoring surface

A good skill is short. The guidance is blunt about it: assume the model is already smart,
keep the body under 500 lines, and when it grows past that, split the overflow into
`references/` one hop away rather than padding the always-competing body. Match the
skill's freedom to the task. Where many approaches are valid, write prose and let the model
choose. Where the work is fragile or must come out identical every time, ship a script and
narrow the model to running it. Anthropic frames the two as an open field versus guardrails
on a narrow bridge. Make the intent unmistakable in the wording, too: "Run
`analyze_form.py`" tells the model to execute, "See `analyze_form.py`" tells it to read,
and confusing the two wastes a turn.

reference and the script live at level three, one read on demand and one executed so the

different relationships with the context window.

One more habit separates skills that work from skills that look like they should. Build the
evaluation first. Run the task without the skill and note exactly where the model fails,
capture that gap into the leanest instructions that close it, and only then iterate. The
practiced version uses two instances: one Claude drafts and refines the skill, a fresh
Claude runs it cold on real tasks, and the second one's failures are the next edit. A skill
you tuned against your own expectations is a skill that fires for you and no one else.

## Install only what you trust

A skill can direct an agent to execute arbitrary code after it triggers, which makes an
installed skill a piece of software you have chosen to run, not a document you have chosen
to read.
Anthropic's guidance is exactly that: use skills only from trusted sources, ones you wrote
or obtained from a party you trust, and audit every bundled file for network calls or file
access that does not match the stated purpose. A skill that fetches an external URL can pull
fresh instructions into the loop on every run, which is the [indirect injection](/mcp-security-surface)
problem wearing a new coat, and a trusted skill can be compromised when its dependencies
change under it. This is why the frontmatter forbids angle-bracket content in the
description: because that string is injected straight into the system prompt, an XML tag
there is an injection surface, and researchers at Oasis Security demonstrated a claude.ai
chain that used a skill to exfiltrate data through the one network endpoint the sandbox left
open. The sandbox is a convenience boundary, not a security one.

<Callout kind="danger">
Treat installing a skill like installing a dependency, because it is one. Read every file
in the folder before you enable it, especially any script that touches the network or the
filesystem, and be doubly wary of a skill that fetches remote content at runtime. In
production, gate code execution and network egress, scope the skill's tools to the minimum
it needs, and back the whole thing with the architectural defenses from [The MCP Security
Surface](/mcp-security-surface). A skill's trust boundary is the code you place on the
agent's own machine, so the only safe default is a source you can vouch for.
</Callout>

Do not bake platform quotas, beta headers, or caching behavior into a portable skill. Pin
those details to the target surface's current documentation and revisit them at upgrade
time. The deeper comparison, when the gap you are filling is procedural knowledge versus
live connectivity, is the next chapter, Skill or Server. Mahesh Murag's one-liner is the
seed of it: MCP provides secure connectivity to external software and data, while skills
provide the procedural knowledge for using those tools effectively. The two compose. A
skill that orchestrates MCP tools is a common production shape.

## Run it

The artifact is a skills lab with no dependencies, no API key, and no network. It ships a
real, correct skill, `changelog-entry/`, that a compatible local skill directory can
discover and trigger, complete with a `references/FORMAT.md` read one hop away and a
`scripts/` validator the skill runs in a feedback loop. Alongside it, `skills_lab.py` does
three things the chapter describes. It validates a portable frontmatter subset: scalar
metadata, including folded or literal descriptions, against the implemented compatibility
rules for name length and character set, the "claude"/"anthropic" ban, angle brackets,
and the 500-line body budget. It also applies explicit authoring heuristics for
third-person phrasing and a concrete `when` cue. The latter is advisory, because it
improves discovery rather than deciding whether the file is structurally valid. It
computes the progressive-disclosure cost, showing what a single skill spends at each level
and what a hundred of them cost at startup. And it simulates discovery, matching a request
against the description and printing the loading sequence that follows. A deliberately
broken `bad-skill/` is included so you can watch representative violations fail, and
`--test` asserts that the good skill passes, the broken skill fails its stated rules, and
synthetic cases cover the numeric boundaries, a folded description, and ordinary
first-person wording.

## Exercises

<ExerciseCard n="01" title="Write the description first">
Take a workflow you repeat, a release note, a code review checklist, a data export, and
write only the `description` field for a skill that would do it. Third person, one to two
sentences, both what it does and the exact words a task needing it would contain. Then run
`python3 skills_lab.py --simulate "<a request in those words>"` and see whether your
description would have been discovered. Tune the words, not the body.
</ExerciseCard>

<ExerciseCard n="02" title="Price the context">
Run `python3 skills_lab.py --budget 100`. Read the startup cost of a hundred skills against
the cost of one activated skill's body. Now state, in one sentence, why installing a large
skill library is cheap but triggering the wrong heavy skill on every turn is not. This is
the difference between level one and level two.
</ExerciseCard>

<ExerciseCard n="03" title="Break a rule and watch it fail">
Open `bad-skill/SKILL.md` and read why it is broken: a name with a forbidden word, an
angle bracket in the description, a first-person voice, and no useful `when` clause. Run
`python3 skills_lab.py --validate bad-skill`, then fix one violation and re-run. Each fix
should silence exactly one finding. The test mode covers the length boundaries separately,
which is what a frontmatter validator buys you before a skill ever reaches a model.
</ExerciseCard>

<ExerciseCard n="04" title="Draw the trust boundary">
Look at the `changelog-entry` skill's `scripts/validate_entry.py`. It touches no network
and no files outside its input. Now imagine a version that fetched the format spec from a
URL at runtime. Write down what an attacker who controlled that URL could make the skill
do, and which of the defenses from [The MCP Security Surface](/mcp-security-surface) would
have to be in place before you would run it.
</ExerciseCard>
