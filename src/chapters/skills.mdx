A skill is a folder that packages procedural knowledge as an on-demand capability. Its
contract is a Markdown file named `SKILL.md`: a few lines of YAML describe the capability,
then a body tells an agent how to use it. Optional scripts, reference documents, and
assets live beside it. The interesting property is not the file format. It is the loading
discipline: an agent can see enough metadata to route work without paying to carry every
workflow in its context window.

The `SKILL.md` convention originated with Anthropic's Agent Skills and became an open
standard in December 2025. The portable core is deliberately small. Individual harnesses
decide discovery, explicit invocation, permissions, and exactly how long loaded content
remains resident. This chapter uses Anthropic's documented implementation for concrete
numbers, but the design pattern is vendor-neutral: [MCP](/resources-tools-and-prompts)
connects an agent to tools and data; a skill supplies the procedure for using them.

## A folder, a file, and a description

The definition is deliberately minimal: a skill is a directory containing a `SKILL.md`
file, optionally alongside bundled scripts, reference documents, and assets. The
`SKILL.md` begins with YAML frontmatter carrying two required fields and continues as an
ordinary Markdown body.

```markdown
---
name: changelog-entry
description: Formats and validates a Keep a Changelog entry. Use when adding a
  changelog entry, recording a change, or editing CHANGELOG.md.
---
# Changelog entry

Add a single line under the Unreleased section. Pick a type, write one
imperative line, then run the bundled validator before you write it.
```

At the portable core, `name` is the identifier and `description` is the routing contract.
The Agent Skills specification requires a one-to-64-character name of Unicode lowercase
alphanumeric characters and single hyphen-separated words that exactly matches its parent
directory. A
name cannot begin or end with a hyphen or contain two consecutive hyphens. It requires a
nonempty description of at most 1,024 characters, and recommends that the description say
both what the skill does and when to reach for it. Anthropic adds surface-specific rules:
it rejects reserved vendor words in `name` and XML tags in `name` and
`description`, and its authoring guidance prefers third-person descriptions. Keep those
layers distinct. A
harness may also expose the name as an explicit command.
By convention `scripts/` holds executable code for deterministic work, `references/`
holds documents the agent reads only when it needs them, and `assets/` holds templates or
fonts used in the skill's output. The recurring analogy is a new-hire onboarding guide:
not a rewrite of the employee, a packet of the context and procedure the job needs. One
general agent plus a library of capability packages, rather than a different architecture
per domain.

## Progressive disclosure

The loading model is the design, and it has a name: progressive disclosure. Content
reveals itself in three levels, each loaded at a different moment and costing the context
window a different amount.

In Anthropic's model-invoked implementation, level one is the listing that the harness
actually gives the model, not a count of folders on disk. In the default full Claude Code
listing, the `name` and `description` of each listed, model-invocable skill enter the system
prompt at startup. A Claude Code skill with `disable-model-invocation: true` is user-only,
so it is absent from that regular-session listing until the user invokes it. Claude Code also
has a `skillOverrides: "name-only"` setting for eligible non-plugin skills: the name stays
listed, but the description does not. When the listing exceeds its budget, every name remains
while low-priority descriptions can be shortened or dropped. The actual post-budget listing
therefore determines both startup cost and which description keywords are available for
discovery.

The next arithmetic scopes itself to a default full name-and-description listing before that
budget reduction. Anthropic's documentation puts that listed metadata near a hundred tokens
per skill. Firecrawl reports a lower external estimate, around thirty to fifty tokens each,
in [Firecrawl](https://www.firecrawl.dev/blog/agent-skills), so one hundred listed,
model-invocable skills with full descriptions cost roughly three to five thousand tokens at
session start. A name-only or budget-trimmed listing costs less, but has less description
surface through which the model can route work. In a regular Claude Code session, level two
adds the full `SKILL.md` body for a first,
distinct, or changed rendering. An identical re-invocation adds a short already-loaded
note instead. Distinct activated skills can stack, while exact re-invocation behavior
remains host-specific. A Claude Code subagent
configured with eligible named preloaded skills takes the other path: it receives those full skill
contents at startup, before a matching task needs to invoke them. The
under-five-thousand-token and 500-line targets are authoring recommendations, not grammar.
At level three, a reference document enters context only when the agent reads it; a script
can execute and return output without first loading its source. Unread resources and
uninspected script source remain on disk, which makes the bundle effectively unbounded.
Anthropic's shorthand for the discipline is that "the context window is a public good," a
principle [Context-Window Economics](/context-window-economics) makes the master
constraint of the whole book.

Do not turn that economy into a universal Skills-versus-MCP rule. Context cost depends on
the host and its configuration. Claude Code now loads MCP tool names at startup and defers
full schemas until a tool is needed, while GitHub's MCP server lets an operator narrow the
available surface with `--toolsets` or `--tools`. An eagerly injected, large tool catalog
is an expensive configuration, not the normal architecture of every MCP host. In a
filesystem-and-shell harness, including Anthropic's illustrated implementation, progressive
disclosure trades startup tokens for retrieval round trips: the agent can use a shell read
to load a body and any referenced file. The portable pattern prescribes staged loading, not
that particular tool or protocol. MCPJam argued that this style of progressive retrieval can
take roughly three to four times the tool calls of loading everything upfront in its
illustrated comparison. Which trade wins depends on the harness, latency budget, and task.
The next chapter, [Skill or Server](/skill-or-server), takes up that comparison; hybrids are common.

## The description does the discovering

Skills are often model-invoked. A harness exposes the actual listing, then the agent decides
whether a task warrants opening a body, much as it decides which tool to call. In a full
listing the `description` does the discovering: when a task matches it closely enough, the
harness loads the full `SKILL.md`; when it does not, the body stays on disk. A name-only
override can still expose a skill's name, but it removes its description from that routing
surface. A listing budget can leave the same partial result at scale. Everything hangs on
this small routing contract, which makes it the highest-leverage line in the file.

The portable specification says the description should state both what the skill does and
when to use it, with words a matching task would contain. Anthropic's authoring guidance
also prefers third person because its description is injected into the system prompt as
fact about the world: "Formats and validates a changelog entry," not "I can help you
format." That is strong surface guidance, not portable validity. Anthropic's current
[`skill-creator`](https://github.com/anthropics/skills/blob/main/skills/skill-creator/SKILL.md)
makes a narrower, Anthropic-specific claim: a simple one-step request such as reading a
PDF may not trigger a skill even when its description matches, while a matching complex,
multi-step, or specialized request reliably triggers. Treat that as current guidance for
that model and harness, not a portable Agent Skills guarantee. On a target harness, use
fresh sessions with realistic positive prompts and near-miss negatives to measure
activation as well as output quality.

Model invocation is not the only door. Some surfaces offer an explicit command for
user-invocable skills, and some offer controls that make a skill user-only or model-only.
Claude Code's `disable-model-invocation` and `user-invocable` frontmatter fields are useful
examples. The first hides a user-only skill from the model's startup listing, which removes
its startup context cost until a user invokes it; `user-invocable: false` hides a model-only
skill from the `/` menu. They are surface extensions, not portable assumptions. Design the
body so either route leads to the same workflow, then test both routes where the target
harness supports them.

## Code that can stay out of the window

The third level is where skills stop being clever prompts and become something a prompt
cannot be. In a harness with a filesystem and a shell, a bundled script can be executed
rather than read. When the model runs `validate_entry.py`, the harness can return its
output without first loading the script's source into context. The agent can still inspect
that source when it needs to, which is why audit remains mandatory. The reasoning is
economic and it is about correctness. Sorting a list by generating tokens is far more
expensive, and far less reliable, than running a sort. Anywhere a task is deterministic,
shipping the code beats asking the model to reconstruct it every time. Willison's
`slack-gif-creator` skill shows the pattern at its cleanest: the model runs a bundled
builder, then a bundled validator checks the result against Slack's file-size limit, and
if it fails the model can adjust and run it again. A feedback loop the skill author wrote
once, executed deterministically, correcting the model's non-deterministic output.

The caveat that bites in practice is that a portable folder does not create a portable
execution environment. Network egress, package installation, filesystem reach, and
available binaries belong to the harness and its policy. A skill that quietly assumes it
can `pip install` a dependency works in one place and fails in another. Start from the
smallest dependency set, state every requirement, and fail clearly when the environment
cannot provide one. Then test the actual surfaces you intend to support.

## The authoring surface

A good skill is short. The guidance is blunt about it: assume the model is already smart,
keep the body under 500 lines, and when it grows past that, split the overflow into
`references/` one hop away rather than padding the always-competing body. Match the
skill's freedom to the task. Where many approaches are valid, write prose and let the model
choose. Where the work is fragile or must come out identical every time, ship a script and
narrow the model to running it. Anthropic frames the two as an open field versus guardrails
on a narrow bridge. Make the intent unmistakable in the wording, too: "Run
`analyze_form.py`" tells the model to execute, "See `analyze_form.py`" tells it to read,
and confusing the two wastes a turn.

loads. Then select a bundled reference or script to see its level-three behavior and the
level-two body directive that points to it. The description sits at level one only when the
actual regular-session listing includes it. In a default full Claude Code listing it does;
a `name-only` override or listing overflow can leave only the name. In a regular Claude Code
session, its full body waits at level two for a first, distinct, or changed rendering; an
identical re-invocation gets a short already-loaded note. A subagent configured with this
eligible skill preloaded starts with its full content instead. Distinct skill bodies can
coexist. The reference and the script live
at level three: reading the reference brings its text into context, while executing the
script can return output without loading source first. One package, five elements, five
different relationships with the context window.

One more habit separates skills that work from skills that look like they should. Build the
evaluation first. Run the task without the skill and note exactly where the model fails,
capture that gap into the leanest instructions that close it, and only then iterate.
Anthropic describes a two-instance workflow: one Claude drafts and refines the skill, a
fresh Claude runs it cold on real tasks, and the second one's failures are the next edit.
A skill you tuned against your own expectations is a skill that fires for you and no one
else.

## Install only what you trust

A skill is an installed capability package, not merely a script. Its `SKILL.md`
instructions, scripts, references, assets, and remote dependencies can direct an agent to
use host tools. Its effective trust boundary is the complete bundle, any content or
dependencies it fetches, and the filesystem, egress, tool, and permission capabilities the
host grants.
Anthropic's guidance is exactly that: use skills only from trusted sources, ones you wrote
or obtained from a party you trust, and audit every bundled file, including `SKILL.md`,
scripts, images, and other resources, for behavior that does not match the stated purpose.
A skill that fetches an external URL can pull fresh instructions into the loop on every run,
which is the [indirect injection](/mcp-security-surface) problem wearing a new coat, and a
trusted skill can be compromised when its dependencies change under it. On a local
filesystem host, bundled code runs with the access that host grants. API Skills instead run
in a sandboxed container with no network access. The environment changes the available
attack paths, not the need for explicit egress, filesystem, and permission controls.
On Anthropic's surface, frontmatter metadata enters the system prompt. The platform
separately disallows XML tags in `name` and `description`. That is not a
portable Agent Skills rule. Separately, Oasis Security demonstrated a Claude.ai chain with
an injection hidden in a URL parameter, then used intentionally permitted Files API egress
to exfiltrate data. It involved no skill, integration, external or user-installed tool, or
MCP server. The lesson is narrower and stronger: a sandbox alone cannot stop prompt
injection or egress it deliberately permits.

<Callout kind="danger">
Treat installing a skill like installing a dependency, because it is one. Read every file
in the package before you enable it, including `SKILL.md`, scripts, references, images, and
other assets, and be doubly wary of a skill that fetches remote content at runtime. In
production, gate code execution and network egress, scope the skill's tools to the minimum
it needs, and back the whole thing with the architectural defenses from [The MCP Security
Surface](/mcp-security-surface). A skill's effective trust boundary is the full package, its
remote dependencies, and the capabilities its host grants. On a local filesystem host, that
can include local-machine access; on a constrained API container, it can be much narrower.
</Callout>

Do not bake platform quotas, beta headers, or caching behavior into a portable skill. Pin
those details to the target surface's current documentation and revisit them at upgrade
time. The deeper comparison, when the gap you are filling is procedural knowledge versus
live connectivity, is the next chapter, [Skill or Server](/skill-or-server). Mahesh Murag framed the split
succinctly: MCP connects agents to external software and data; skills provide the
procedural knowledge for using those tools effectively. The formulation comes from a
[VentureBeat interview](https://venturebeat.com/technology/anthropic-launches-enterprise-agent-skills-and-opens-the-standard).
MCP does not make a connection secure by itself. Its authorization framework is optional;
authentication, transport, authorization, and least-privilege deployment determine whether
an integration is secure. The [MCP authorization specification](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization)
and [security best practices](https://modelcontextprotocol.io/docs/tutorials/security/security_best_practices)
name the required implementation work. The two compose. A skill can orchestrate MCP tools
when it supplies the procedure those tools need.

## Run it

The artifact has no third-party packages, API key, or network requirement. It needs a
`python3` command backed by Python 3.9+, plus Bash for its documented install and check
paths. It ships a real, correct skill, `changelog-entry/`, that a compatible local skill
directory can discover and trigger, complete with a `references/FORMAT.md` read one hop away
and a `scripts/` validator the skill runs in a feedback loop. If you create
`~/.claude/skills/` after a Claude Code session begins, restart the session so it watches the
new top-level directory. Additions below an existing watched directory take effect live.
Alongside it, `skills_lab.py` does
three things the chapter describes. Its default teaching lint checks the bundle's supported
portable subset: ASCII name syntax, directory matching, and description bounds. It rejects
malformed or unsupported YAML forms instead of reporting malformed frontmatter as clean. It
is not a general YAML or full-schema validator. For production, use the target
harness's maintained validator and a deployment gate. `skills-ref` is a demonstration-only
reference implementation, useful for comparison but not production certification. An explicit
Anthropic profile adds that surface's reserved-word and XML-tag checks. Third-person
phrasing, a concrete `when` cue, and the 500-line budget remain labelled authoring warnings,
not portable failures. The lab models the actual regular-session listing: `--budget 100`
starts with the default full name-and-description listing; `--listing name-only` removes
descriptions; and `--descriptions 20` models a budget-trimmed listing where all names remain
but only twenty descriptions reach the model. The last two cases reduce both startup tokens
and description-driven discovery. The lab makes the user-only exception visible and offers a
preloaded-subagent mode where eligible named full skill bodies arrive at startup instead.
Its discovery simulator is deliberately illustrative: it uses a two-keyword proxy and
prints a possible loading sequence, not evidence that a production model would trigger. A
deliberately broken
`bad-skill/` is included so you can watch the layers fail, and `--test` exercises the
portable boundary cases, the surface profile, raw stdin validation, terminal-control
rejection, the installed-validator path, literal shell-syntax preservation, and both
lifecycle modes.

## Exercises

<ExerciseCard n="01" title="Write the description first">
Take a workflow you repeat, a release note, a code review checklist, a data export, and
write only the `description` field for a skill that would do it. One to two sentences,
both what it does and the exact words a task needing it would contain. Third person is good
Anthropic authoring guidance, not a portable validity rule. Then run `python3 skills_lab.py --simulate "<a request in those words>"` as an illustrative keyword proxy.
For evidence, use a fresh model in the target harness with realistic positive prompts and
near-miss negatives. Tune the words, then measure the actual trigger behavior.
</ExerciseCard>

<ExerciseCard n="02" title="Price the context">
Run `python3 skills_lab.py --budget 100`. That is the default full listing: a hundred
listed, model-invocable names plus their descriptions. Then compare `--listing name-only`
and `--descriptions 20`. State why the actual post-budget listing, not the installed library
count, determines both startup cost and which description keywords can trigger discovery.
Then run `python3 skills_lab.py --budget 3 --session preloaded`. Explain why a large listed
skill library can stay cheap while either triggering the wrong heavy skill or preloading it
into a configured subagent consumes its full body budget. Name the other exception: a Claude
Code user-only skill with `disable-model-invocation: true` is absent from regular-session
startup context until manual invocation.
</ExerciseCard>

<ExerciseCard n="03" title="Break a rule and watch it fail">
Open `bad-skill/SKILL.md` and read why it is broken. Its name violates the portable syntax
and does not match its directory. Its description also carries an Anthropic-only XML-tag
restriction and weak authoring guidance. Run `python3 skills_lab.py --validate bad-skill`, then add `--surface anthropic` to see the surface profile. Fix the
portable name and directory together first; then decide which harness-specific guidance you
need. The test mode covers the boundary cases a validator buys you before a skill reaches a
model.
</ExerciseCard>

<ExerciseCard n="04" title="Draw the trust boundary">
Look at the `changelog-entry` skill's `scripts/validate_entry.py`. The validator touches no
network and does not write project files. The broader skill intentionally writes
`CHANGELOG.md` after validation, so it needs that project-write permission. Now imagine a
version that fetched the format spec from a URL at runtime. Write down what an attacker who
controlled that URL could make the skill do, and which of the defenses from [The MCP
Security Surface](/mcp-security-surface) would have to be in place before you would run it.
</ExerciseCard>
