An agent is a loop. Not a model, not a prompt, not a framework: a loop that calls a
model, runs whatever the model asked for, feeds the result back, and calls the model
again until there is nothing left to do. The word "agent" names that cycle, not the
network inside it. This is the whole book compressed into one sentence, and every
later chapter is an argument about how to build the loop well: what to put in the
context, when to stop, how to remember, when to split the work. The shape is
provider-neutral. Specific APIs enter only where they make a concrete contract visible.

## The loop is the primitive

The cycle has four phases. **Perceive**: the harness assembles a prompt from the
system instructions, the tool definitions, and the running message list. **Decide**:
the model runs one forward pass and emits text, tool-use requests, or both. **Act**:
the harness parses the requests, checks permissions, and runs the real tools.
**Observe**: the tool results are appended to the message list, and the cycle repeats.

This is an old shape wearing new clothes. It descends directly from sense-plan-act,
the deliberative architecture behind Shakey the robot at Stanford Research Institute
in the late 1960s: perceive the world, update a model of it, plan, act, repeat. The
difference is what fills the boxes. The "world model" is now the accumulated text in
the context window, and the "planner" is a frozen neural policy that you cannot
retrain in the loop. Anthropic gives a compact version of the definition: agents
typically use tools based on environmental feedback in a loop. The decide phase is
where reasoning and action interleave, the pattern Yao and colleagues named ReAct in
2022.

## The model is a stateless policy

The loop gives the model no implicit continuity between calls. Treat the next response
as a sampled action from the context the harness assembles this turn. Everything that
feels like memory, continuity, or progress therefore lives in one place: the running
message list. That list is the agent's working memory. The system prompt, the tool
schemas, the conversation, and every tool result share a single context window, and
the list grows in exactly two places. It grows on **decide**, when the assistant's
turn is appended, and on **observe**, when the tool result is appended. Perceive only
reads the list; act only touches the outside world.

Because the policy is frozen, the lever you actually control is the context you
construct: the environment, the tools, and the system instructions. Here is the loop
with nothing removed. Its control flow is provider-neutral; the block names below are
Anthropic Messages API terms because the runnable artifact uses that API as one
concrete adapter.

```python
from anthropic import Anthropic

client = Anthropic()
messages = [{"role": "user", "content": "What files are here?"}]

while True:
    reply = client.messages.create(
        model="claude-sonnet-5", max_tokens=1024,
        tools=TOOLS, messages=messages,
    )
    messages.append({"role": "assistant", "content": reply.content})  # verbatim
    if reply.stop_reason != "tool_use":
        break
    results = [
        {"type": "tool_result", "tool_use_id": b.id, "content": run_tool(b.name, b.input)}
        for b in reply.content if b.type == "tool_use"
    ]
    messages.append({"role": "user", "content": results})
```

Two API invariants hold this together. Append the assistant turn verbatim, so the
`tool_use` IDs the model generated stay aligned with the results you send back. And
put the `tool_result` blocks in a user message that immediately follows their
`tool_use` blocks. Break either invariant and the next call rejects the history. Note
what is absent: there is no planner, no memory store, no orchestration graph. The
`while` is the agent.

## Termination belongs to the harness

Look again at the one branch in that loop: the simple client-tool loop exits when
`stop_reason` is not `tool_use`. Exiting the tool loop is not always successful
completion. Deciding what to do next is a harness responsibility, and it has to be,
because models are unreliable at knowing when they are done. Left alone, an agent will
declare victory early or grind in circles late. So the harness owns the stop
conditions, and there are five worth naming: the model requests no tool, an explicit
completion signal fires, a hard iteration limit is reached, a cost or token or
wall-time budget is spent, or an error state is hit. The Anthropic Messages API
surfaces several of these as `stop_reason`: `tool_use` means the model wants a tool,
`end_turn` usually means it finished, while `max_tokens`, refusal, and context-window
limits need their own policy. A production loop wires the other conditions around that
signal as ordinary control flow.

## Minimal is startlingly strong

The unsettling part is how little you need. mini-swe-agent, from the SWE-bench team at
Princeton and Stanford, is roughly a hundred lines of Python. It uses only bash, no
tool-calling API at all, so it runs with any model, and its maintainers report a score
above 74 percent on SWE-bench Verified. That is a dated, model-and-harness-specific
result, not a permanent leaderboard claim. Still, a hundred lines of loop can be
competitive on a hard real-world coding benchmark. The team's own conclusion is blunt:
"the interface is the model." When the harness is thin, the score comes closer to a
measurement of the model, which is exactly why SWE-bench fixes a minimal harness to
compare models fairly.

The same lesson shows up across coding-agent implementations. A small reactive loop
sits under a much larger operational harness: permissions, compaction, steering,
persistence, and verification. The intelligence comes from the model; the reliability
comes from the system that contains it.

This does not mean scaffolding is worthless. Tool interfaces can materially improve
results. In one dated GPT-4 Turbo comparison on SWE-bench Lite, the original SWE-agent
paper reported a 64 percent relative gain over its shell-only baseline. As base models
strengthen, the marginal value of clever scaffolding shrinks, while harness engineering
still decides reliability, cost, and safety. Build the loop first. Add harness, not
decoration.

That also tells you when to reach for an agent at all. A **workflow** runs LLMs
through control flow you wrote in advance; you own the path. An **agent** lets the
model direct its own tool use; the model owns the path. If you can draw the decision
tree, write the workflow. Reserve the agent for tasks that are ambiguous enough that
you cannot pre-map the path, valuable enough to justify the token spend, and forgiving
enough that a wrong step is detectable or cheap. Coding fits all three, which is why it
became the exemplary agent use case.

## Where loops break

If the loop is this simple and this strong, the rest of the book is about the ways it
fails at scale. Three failures are structural and set up everything that follows.
Context accumulates: every turn adds tokens, and tool outputs balloon fastest, so an
agent can run out of room after only a handful of calls. Errors cascade: one
wrong step feeds the next, and per-step error rates multiply across a long chain.
Coherence decays: attention degrades over long contexts, the lost-in-the-middle
effect, so a model can forget an instruction it was given while the window is still
half empty.

Each failure has a chapter aimed at it. Context economics is the master constraint of
Part I. Memory, in Part IV, is the set of answers to accumulation and decay. Splitting
work across isolated agents, in Part III, is one answer to cascades, and its own
source of new failures. All of it is engineering on top of the loop you just built.

In `loop.py`, add a `search(pattern)` tool that greps the files in the directory, and
change the task to something that needs it (for example, "which file defines

<ExerciseCard n="02" title="Give the harness a budget">
Add a hard cap on total tool calls, separate from `MAX_TURNS`. When the cap is hit,
append a `tool_result` that says so and let the model wrap up. Confirm the model cannot
spend past the budget. Termination is yours, not the model's.
</ExerciseCard>

<ExerciseCard n="03" title="Break termination on purpose">
Comment out the `stop_reason != "tool_use"` check and give the agent a task that needs
no tool. Watch it run to `MAX_TURNS` instead of halting. This is why the stop condition
lives in the harness: without it, the loop never ends on its own.
</ExerciseCard>

<ExerciseCard n="04" title="Watch context grow">
Print an estimated token count on every perceive phase (a rough proxy is
`len(json.dumps(messages)) // 4`). Run a few turns and watch it climb monotonically.
That curve is the subject of the next chapter on context-window economics.
</ExerciseCard>
