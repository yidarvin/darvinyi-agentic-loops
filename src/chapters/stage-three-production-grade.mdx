Stage Two makes an agent durable inside one process. Stage Three makes that loop useful and safe beyond the demo: one dispatch core gains MCP clients for extensibility, isolated subagents for delegation, durable memory for continuity, and a sandboxed execution boundary for safety. The model still chooses. The harness now constrains what its choices can reach.

## The robust loop has four missing boundaries

A robust loop can retry, stream, validate tool inputs, and compact context, yet still be closed, single-agent, amnesiac, and unconfined. Those are separate failure classes. Adding an MCP client lets the agent consume external capability without hand-writing every integration. Durable project context prevents every session from rediscovering the same conventions. A subagent gives a noisy inspection task its own context window. A sandbox limits the blast radius when any tool, model output, package script, or retrieved document is wrong.

Do not turn this into a feature checklist. Each capability belongs at a precise seam in the existing loop. Startup loads project context and selected memory. Tool discovery merges native and MCP-backed tools into one exposed registry. Dispatch maps an exposed tool name back to its implementation. A subagent call is the same loop with a fresh conversation and a smaller tool set. Each launched MCP or shell process crosses policy before the operating-system boundary. Host-owned startup memory I/O needs its own descriptor-contained path.

The distinction at the bottom of the figure should survive design changes. Permissions answer, “should this run?” A sandbox answers, “if it runs, what can it touch?” The first is an intent policy. The second is containment. They complement each other; neither replaces the other.

## Make MCP tools ordinary dispatch entries

An agent is an MCP host. It owns one client session per configured server, initializes each transport, discovers tools, and maps every external definition into the same tool shape that the model already sees. With more than one server, expose a stable namespace such as `mcp__git__status` and retain a reverse map to the client session and original tool name. The model gets a flat menu. The dispatcher keeps the provenance.

That separation is what keeps the Stage Two loop intact. MCP does not need a second reasoning path. It needs connection lifecycle, definition caching, timeouts, graceful degradation when a server dies, and tool-definition drift detection. The [MCP architecture guide](https://modelcontextprotocol.io/docs/learn/architecture) describes the host, client, and server roles along with the `initialize`, discovery, and invocation lifecycle.

Treat server metadata and results as untrusted input. A tool description can contain instructions, and a previously benign definition can change after approval. Pin normalized tool definitions by hash, display the exact arguments that will run, isolate credentials per server, and run local servers inside the same containment boundary as the rest of the agent. The [MCP security guidance](https://modelcontextprotocol.io/docs/tutorials/security/security_best_practices) is the protocol-level starting point, not a substitute for a host threat model.

## Delegate to isolate context, not to multiply writers

A subagent is a recursive call to the loop with a task string, a clean conversation, a restricted tool set, and a depth counter. The parent sends the minimum context required to do the task. The child returns a bounded final summary, not its transcript. That is context management by isolation: a worker can read forty files and run a noisy search without forcing those details into the coordinator's window.

Keep the first version narrow. Set `MAX_DEPTH = 1`, remove the spawn tool from workers, make exploratory workers read-only, and let one coordinator own writes. Parallelize only independent, read-heavy tasks. Dependent edits and shared mutable state need a single linear owner. Anthropic's reported multi-agent token multipliers are directional internal measurements, not a benchmark guarantee, but they reinforce the engineering rule: buy extra loops only when their isolated result changes a valuable decision. See the clearly labeled product example, [Anthropic's multi-agent research system](https://www.anthropic.com/engineering/multi-agent-research-system), for a production account of that tradeoff.

## Persist facts, then defend the persistence layer

Start with a short project context file that the agent reads at startup. It should hold stable commands, local conventions, and decisions that would otherwise be repeated. When that is insufficient, add a file-backed memory tool with explicit `view`, `create`, and replacement operations. Resolve every requested path and prove it remains under the memory root before opening it. String-prefix checks fail on traversal and symlink cases.

Memory also changes the injection model. A bad web page affects one turn. A poisoned memory file reloads on every future turn. Keep memory per user or project, bound and prune it, filter secrets before writing, and treat stored text as data rather than privileged instructions. Claude Code's [memory documentation](https://code.claude.com/docs/en/memory) is a product example of layered project instructions and local memory, not a universal storage design.

## Put policy in front of a real boundary

The permission evaluator should be simple enough to audit: evaluate `deny`, then `ask`, then `allow`; use first match within a tier; never let an allow rule override a deny. Deny reads of secrets, destructive shell patterns, and unbounded network egress. Ask before irreversible state changes. Allow the small read-only set that makes normal work quiet. Log every decision and every tool invocation.

<Callout kind="danger">
An allowlist and an approval dialog do not confine a compromised process. Run bash, local MCP servers, package scripts, and their children in an OS sandbox or stronger isolated environment. Strip credential-shaped environment variables, scope filesystem writes to the workspace, restrict egress, and keep a human approval step when an operation would complete the private-data, untrusted-input, and external-communication trifecta.
</Callout>

For macOS, Seatbelt can enforce filesystem and network rules at process launch. On Linux, a comparable layer might use bubblewrap or a container. For higher-assurance workloads, use a VM or microVM boundary. Anthropic reports that its sandboxing reduced permission prompts while users approved most prompts in its internal data. Those measurements are useful directionally, not independently verified security guarantees. The labeled product example is [Anthropic's sandboxing report](https://www.anthropic.com/engineering/claude-code-sandboxing).

## Build the harness in the order that preserves control

Put the sandbox and deny-first policy in place before adding more power. Then connect one trusted MCP server over a local transport, namespace its tools, and record its definitions. Add a compact project context file when the agent repeats work across sessions. Add one read-only subagent only when a task produces enough disposable output to contaminate the parent context. Install token and depth limits before allowing parallel work.

The runnable artifact makes these seams observable without an API key. It is a deterministic harness probe, not a claim that a scripted planner is a general coding model. It authorizes a local stdio MCP server before launch, streams structured events, pins its tool definition in host-owned state, loads memory through a contained path resolver, delegates a read-only subagent, and routes a verification command through deny, ask, allow policy and macOS Seatbelt. Outside macOS, execution fails closed rather than silently running unsandboxed.
