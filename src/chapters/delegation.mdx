Without compaction or externalized memory, a single-agent loop has one working context
window. Every file body, failed command, and long tool result it keeps in that history
consumes the same window. Delegation is one context-management strategy, not the only
structural answer. Compaction preserves a continuous exchange by distilling history;
structured note-taking preserves selected state outside the window; delegation gives
focused, independently explorable work its own loop. The reflex is to reach for delegation
when you want parallelism, and parallelism is real, but it is the smaller half of the
story. For a self-contained slice that can return a bounded result, delegation first
decides which work burns which window, and only then does it happen to run several windows
at once. Leave a breadth-first task in one uncompacted window and its history fills with
noise until relevant facts are buried and its answers degrade.

This chapter opens Part III by establishing the mechanic and nothing more. The topologies
that connect many agents belong to [Coordination Patterns](/coordination-patterns), and the
ways they fail belong to [When Multi-Agent Fails](/when-multi-agent-fails). Here we build
the atom those chapters compose: one lead, one isolated subagent, a prompt down and a
result up.

## The ceiling a single window hits

[Context-Window Economics](/context-window-economics) set the budget. This is the failure
mode that spends it. As a window fills, its signal-to-noise ratio collapses, and the model's
answers get worse even though the weights never changed. Three mechanisms compound. Liu and
colleagues named the first in "Lost in the Middle": accuracy is U-shaped by position. In
their GPT-3.5-Turbo multi-document QA evaluation, performance can drop by more than twenty
percent in the worst case when the relevant document is poorly placed. The second is
attention dilution, a fixed attention budget spread thinner across every additional token
pair. The third is distractor interference, where similar-but-irrelevant content pulls the
answer off course.

Chroma's 2025 "Context Rot" report broadens the case. Across its experiments, performance
generally degrades as input length grows. It covers eighteen frontier models overall,
including Claude 4 Opus and Sonnet, Gemini 2.5, and Qwen3, but not every model appears in
every experiment and individual curves are not monotonic. GPT-4 Turbo, for example, has a
local performance peak around 500 words in the repeated-words experiment. The report found
no notable needle-position variation for its specific NIAH task. The practical reading is
blunt. A 200,000-token window is not 200,000 tokens of usable working memory. A task that
reads fifty files to answer one question is self-defeating in a single window, because by
the time the answer is due, the files that mattered are buried in the ones that did not.

## A subagent is a tool that happens to be an agent

The mechanic reuses machinery you already have. The lead decomposes the task and spawns a
subagent by emitting a [tool call](/anatomy-of-a-tool-call). In Claude Code and the Claude
Agent SDK that call names the Agent tool, renamed from Task in version 2.1.63, with both
names still accepted. The runtime spins up an isolated agent loop, lets it run to
completion, and returns its final report through the Agent tool-result path. The subagent is
a tool whose implementation is another agent loop. The same shape recurs across stacks: the
OpenAI Agents SDK exposes it as `Agent.as_tool()`, and LangChain handoffs use tools to
update routing state and transfer work. Delegation is not a new primitive. It is the tool
call, pointed at an agent.

What makes it delegation rather than a function call is the context the subagent starts
with. Take a one-shot, non-fork Claude Code delegation with no SendMessage exchange or
resume. Its context is fresh but not empty: it receives its own system prompt and environment
details, the task message, the CLAUDE.md and memory hierarchy, a git-status snapshot, and
any preloaded skills. In configurations that enable it, it can also receive a sibling roster.
It does not receive the parent's conversation, tool results, or full system prompt. The task
message is therefore a task-specific briefing, not the subagent's complete input. Put file
paths, error messages, and decisions that are not already in the runtime baseline into that
briefing. When the worker finishes, its final report becomes the Agent tool result. Claude
Code scans that report before the parent reads it and can add markers or escapes. Resumed and
message-driven workers use additional paths that this one-shot model intentionally excludes.

## What isolation buys

The mess stays on the subagent's side. A research-assistant subagent can read dozens of
files without a line of that content accumulating in the main conversation; the parent
receives a concise summary, not every file the subagent opened. The human analogy is exact.
A manager who delegates does not watch every keystroke. They get the memo. Cognition's
Walden Yan, arguing hard against most multi-agent designs, still concedes the benefit: the
subagent's investigative work does not need to remain in the main agent's history, which
lets the main trace run longer before it runs out of room.

There is a sharper corollary that is easy to miss. A clean context does not only save space,
it can make the subagent smarter. Cognition found their Devin code-review loop works best
when the coding agent and the reviewing agent share no context beforehand. The reviewer
skips the extraneous history, looks only at the diff, and rediscovers whatever it needs by
reading the code from scratch, and with a shorter context the sharper reasoning catches more
nuanced issues. Even on pull requests Devin wrote itself, their reviewer averages about two
bugs caught per PR, roughly 58 percent of them severe. Isolation is not just hygiene. Less
context is sometimes a better context.

None of this is free, and the price is the whole reason to be disciplined about it.
Anthropic measured agents using about four times the tokens of a chat and multi-agent
systems about fifteen times. Their multi-agent research system, a Claude Opus 4 lead with
Claude Sonnet 4 subagents, beat single-agent Opus 4 by 90.2 percent on their internal
research eval and cut research time by up to 90 percent through parallelism. But token usage
alone explained about 80 percent of the performance variance, so much of what you buy, you
buy with tokens. These are Anthropic's own internal numbers on breadth-first research, not
an independent benchmark, and they do not transfer to coding or to simple tasks. The lesson
that does transfer: delegation earns its cost only when the per-task value can absorb a
double-digit token multiple. The canonical win is telling. Asked to name every board member
of the S&P 500 IT companies, the multi-agent system decomposed the query and found the
answers while a single agent ground through the same searches sequentially and failed.

## Teach the orchestrator to delegate

Because a non-fork subagent does not inherit the lead's conversation or tool trace, the
task-specific briefing carries the work it must do, and Anthropic calls writing it well their
single most important lesson. It is not the worker's entire startup context: the runtime
baseline remains separate. Each subagent needs an objective, an output format, guidance on
which tools and sources to use, and clear task boundaries. Skip that and agents duplicate
work, leave gaps, or never find what they were sent for. Their failure example is concrete:
told only to "research the semiconductor shortage," one subagent went after the 2021
automotive chip crisis while two others duplicated each other on the current supply chain,
with no division of labor because none was specified.

Scale the effort to the task, and put the rule in the orchestrator's own prompt so it stops
over- and under-spending. Anthropic's embedded guidance is a good default: simple
fact-finding wants one agent making three to ten tool calls; a direct comparison wants two
to four subagents with ten to fifteen calls each; only genuinely complex research justifies
ten or more subagents with clearly divided responsibilities. That single instruction
corrected both failure directions they saw, from spawning fifty subagents for a trivial
query to cramming a breadth-first task into one overloaded agent.

Specialization is the other dial, and it has four notches: a focused system prompt that
would be noise in the main agent's instructions, a restricted tool set so a read-only
reviewer can analyze but never modify, a cheaper model like Haiku for bulk reading, and a
smaller thinking budget for shallow work. A Claude Code subagent is a markdown file with
YAML frontmatter, and its `description` field is what the lead reads to decide when to
delegate, so that one line does real work. But the first question is always whether to
delegate at all. Keep the work in the main conversation when it needs frequent
back-and-forth, when several phases share significant context, when the edit is small and
targeted, or when latency matters, because a subagent starts cold and pays to re-gather
context. Delegate when the output is verbose and unneeded upstream, when you want tool
restrictions, or when the work is self-contained and can come back as a summary.

## The return is a compression boundary

The down-channel is the task-specific briefing. The up-channel decides whether delegation
paid off. If a subagent returns its full context, you lose the entire benefit: you paid to
read everything, then paid again to copy it into the lead, and polluted the very window you
were protecting. The return has to be distilled. In the one-shot Claude Code model, the
worker's final report becomes the Agent tool result, but the runtime can scan and annotate it
before the parent reads it. Put the compression discipline in the subagent's instructions:
return a structured summary, not a transcript.

For large or high-fidelity outputs, distillation would throw away too much, and the answer
is not to widen the channel but to move around it. Anthropic's pattern is to let subagents
write their work directly to the filesystem and pass back a lightweight reference. The heavy
artifact, a code file, a report, a chart, persists outside the conversation, and only the
pointer crosses the boundary. This avoids the game of telephone that lossy re-summarizing
introduces and the token cost of dragging big outputs through the history. Either way the
principle is the same: the return channel is a compression boundary, and you design it on
purpose.

## Where delegation stops

The mechanic has a hard edge, and it is exactly where the next two chapters begin.
Cognition's "Don't Build Multi-Agents" (June 2025) is the sharpest statement of it, on two
principles. Share context, and share full agent traces rather than isolated messages. And
actions carry implicit decisions, so when agents act in parallel their conflicting decisions
compound into a broken result. Their example sticks: ask parallel subagents to build a game,
one making the background and another the bird, and you get visually incompatible assets,
because neither subagent could see what the other chose.

By 2026 both camps had converged on where the line sits. Cognition's follow-up,
"Multi-Agents: What's Actually Working" (April 2026), keeps the narrower pattern that holds
up: setups where several agents contribute intelligence to a task while the writes stay
single-threaded, a manager that splits work, children that execute, and one thread that
synthesizes and commits. Anthropic and Cognition agree on the safe zone from opposite
directions, and it is precisely the delegation this chapter describes. Reading, exploring,
and analyzing parallelize cleanly, because they make no conflicting decisions. Writing does
not, so funnel every state-mutating action through one thread until models coordinate
write-decisions reliably. That boundary, and the topologies and failure modes on either side
of it, are what [Coordination Patterns](/coordination-patterns) and [When Multi-Agent
Fails](/when-multi-agent-fails) take up next.

## Run it

The artifact makes the boundary observable instead of asserting it. `delegate.py` is a
deliberately narrow lab model, not a full Claude Code or Claude Agent SDK transport model.
It models one isolated, one-shot, non-fork worker with no messages or resume. The lead and
worker are separate context objects. The worker starts with a modeled runtime baseline and
receives a task-specific briefing without the lead context, so isolation is enforced by the
call signature, not drawn as a picture. Run `--compare` and the same task runs three ways:
inline, where the lead reads every file into its own window; delegate, where the modeled
return is a distilled summary; and leak, where the worker returns its full transcript.
Delegation keeps the lead more than thirteen times smaller than inline; leaking the
transcript throws that away and lands right back near inline. `--show-boundary` prints the
lab model's direct briefing and selected return, and `--test` asserts no file body ever
reaches the lead's context. It runs offline with the standard library. With a key, `--live`
makes one Anthropic Messages API call as a live model stand-in. It is not an Agent tool or a
tool-using Claude subagent loop, and it degrades quietly to offline when unavailable.
