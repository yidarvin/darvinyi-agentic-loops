Each model call sees only the context its harness supplies. An agent can still maintain
active state across calls, persist durable records outside that state, and select useful
material for the next call. The taxonomy matters because working, episodic, semantic, and
procedural memory solve different failures. Treating them as one generic vector store
produces stale context, unsafe writes, and agents that remember the wrong thing.

## Working memory is active agent state

The context window is a projection of working memory, not working memory itself. In
CoALA, working memory is the agent's active state across its decision loop: active goals,
variables, recent tool results, and retrieved material. It can persist across several LLM
calls. For each call, the harness synthesizes a context projection from the relevant
subset. A session boundary can reset or compact that state, but a model call is not its
natural lifetime.

This continues the constraint from [Context-Window Economics](/context-window-economics).
The per-call projection is bounded and token-expensive. Every extra memory record competes
with instructions, evidence, tools, and output space. A long context full of weakly
relevant recollections can perform worse than a short one with the right records.

Do not confuse parametric knowledge with agent memory. Model weights may contain broad,
static knowledge learned during training. Agent memory is dynamic external state for this
user, task, or organization. It can be scoped, inspected, updated, and deleted. That
control plane is the reason production agents usually place mutable information outside
the weights.

## Four stores answer four questions

CoALA gives the useful map: one short-term working store and three long-term stores. The
working store holds active agent state, while a context window is the per-call projection
of that state. The boundaries are engineering boundaries, not a claim that cognition has
four sealed boxes. An episode can become a fact through consolidation, and a fact can later
shape a rule. The questions below expose the distinction that matters when designing reads
and writes. Long-term means information retained beyond the active working state.

- **Working memory** answers: what is active in this decision loop? It carries goals,
  variables, tool results, and retrieved material across calls. Each call sees a synthesized
  context projection, not the whole working state.
- **Episodic memory** answers: what happened? It keeps dated interactions, trajectories,
  tool outcomes, and successful examples with their original circumstances.
- **Semantic memory** answers: what is true? It holds decontextualized facts, profiles,
  preferences, and durable knowledge.
- **Procedural memory** answers: how do we act? It includes implicit model weights and
  explicit system prompts, skills, code, and tests. Weights shape every generation without
  retrieval. An explicit skill can be selected or densely retrieved from a code library,
  then added to the context projection or executed by the harness.

Consider three records from a release assistant. "Mira said she eats vegetarian food" is
an episode. "Mira prefers vegetarian food" is a semantic fact distilled from that episode.
"Run migration verification before production deployment" is a procedure when it is
enforced by a release skill, a CI check, or an instruction. The same sentence may pass
through several forms, but each form has a different retrieval path and a different
write-risk profile. Editing a user profile is reversible. Editing a deployment procedure
can change future behavior for everyone.

## Memory is a lifecycle, not a database choice

Every design has the same movement: encode a candidate, store it with metadata, select or
retrieve an external record into working memory, then update, forget, or consolidate it.
The hard decision is not where to put text. It is which candidate deserves a future context
slot.

Write records with a type, scope, provenance, timestamp, and retention rule. Scope is
usually per user or per session. Shared multi-agent memory needs an explicit owner and
writer policy, because one polluted record can contaminate every downstream agent.
Provenance lets an operator trace and remove a bad record. A timestamp lets a system tell
"Mira preferred vegetarian food last year" from a current preference.

Retrieval should follow the question. Vector search works well for fuzzy recall over
episodes or facts. A key-value or document store suits an exact profile. A temporal graph
earns its cost when facts change over time and point-in-time answers matter. Files are
often the simplest durable scratchpad because an agent and an operator can inspect them
directly. None of these substrates make memory useful by themselves. A record still has to
win a retrieval decision and fit the next context window.

Forgetting is equally architectural. A profile may overwrite a superseded address. An
audited system may append a new value and close the old value's validity range. A history
may compress many episodes into one semantic note. The right policy depends on whether the
next question is "what is current?" or "what did we know then?". Start with context
management or direct retrieval from an existing source of truth. Add long-term memory only
when cross-session continuity, personalization, or long-horizon learning needs it.

<Callout>
Persistent memory is a security boundary. [Dong et al.'s MINJA attack](https://arxiv.org/abs/2503.03704)
shows that an attacker can inject malicious records through query-only interaction. In a
design that writes untrusted interaction and later retrieves those records, the poison can
influence future behavior. Namespace memory per user, restrict who can write, keep
provenance, and filter sensitive fields before storage.
</Callout>

## Product examples are implementations, not the taxonomy

**Product examples.** MemGPT and Letta frame the context window and external stores as an
OS-like hierarchy. LangMem exposes episodic, semantic, and procedural memory as first-class
concepts. Mem0 emphasizes extracted user facts, while Zep and Graphiti emphasize temporal
facts and provenance. These are useful designs, but none changes the underlying test:
identify the question, choose the store and lifecycle that answer it, then measure whether
the retrieval earns its context cost.

The evidence should include accuracy, token cost, and latency together. LongMemEval and
LoCoMo exist because retaining information across many sessions remains difficult. Vendor
benchmarks can show a direction, but headline comparisons often include different retrieval
pipelines, embeddings, and judge models. Use them to generate a hypothesis, then test the
memory policy against the agent's own tasks.

<ExerciseCard n="01" title="Classify one agent's state">
Choose an agent you run. List six pieces of state it carries, then label each working,
episodic, semantic, or procedural. For each ambiguous item, state the question it must
answer and the next action that would write it. Do not pick a substrate until the type is
clear.
</ExerciseCard>

<ExerciseCard n="02" title="Make a changing preference auditable">
Design the record for a user preference that can change, such as dietary restrictions or a
deployment window. Specify its namespace, source event, timestamp, current-value rule,
retention period, and the operator action that can remove it. Decide whether update means
overwrite, append, or invalidate.
</ExerciseCard>

<ExerciseCard n="03" title="Run a failed trace through the taxonomy">
Run the harness, replace its deployment incident with a real failed task trace, and rerun
the five regimes. Decide which parts should remain as episodes, consolidate into facts,
become a tested procedure, or be discarded. Defend the procedural write especially.
</ExerciseCard>

<ExerciseCard n="04" title="Put a fence around shared memory">
Sketch a shared store for three cooperating agents. Name the namespace, the single writer

budget per turn. Then name one poison record your design rejects before it reaches another
agent.
</ExerciseCard>
