# Prompt Caching and the Economics of Remembering

Research reference for *Agentic Loops*, Chapter 16 (second chapter of Part IV, Memory; follows "The Memory Taxonomy," precedes "Retrieval as Memory" and "Self-Managed Memory"). Current as of mid-2026. The reader already knows the memory taxonomy (working/episodic/semantic/procedural, short-term vs long-term, substrates, the lifecycle) and, from Part I, context-window economics (token pricing, the quadratic resend cost, context rot). This chapter goes deeper specifically on caching as a memory/cost mechanism. **All pricing, TTLs, minimums, and model names are point-in-time (mid-2026) and change fast — re-verify against official pages before committing budgets.**

## TL;DR
- **Prompt caching is the mechanism that makes "keeping things in context" cheap.** By storing the processed KV (key-value) attention state of a stable prefix, providers let subsequent calls sharing that prefix skip prefill — cutting cached-input token costs by ~90% on Anthropic, OpenAI, Google, and DeepSeek, and reducing time-to-first-token up to ~85% for long prompts. This inverts the naive economics of memory: for a large, stable, frequently-reused body of context, remembering it in-window can be cheaper than retrieving it.
- **Caching is prefix-based and exact-match**, a direct consequence of autoregressive attention: you can only reuse computed state up to the first divergent token. Iron rule: stable content (system prompt, tool definitions, few-shot examples, reference documents) goes at the FRONT; volatile content (latest user turn, freshly retrieved memory, timestamps) goes at the END. Agentic loops are the ideal case because their prefix grows monotonically and their stable head is byte-identical every turn.
- **The right architecture treats caching, retrieval, and fine-tuning/recompute as three levers of the economics of remembering.** Caching makes keeping-in-context cheap when reuse is high and content is stable; retrieval makes on-demand recall cheap when the corpus is large or churns; recompute/summarization is the fallback. Instrument cache-hit rates; treat caching as a monitored feature, not a fire-and-forget flag.

## The core problem: the statelessness tax
Because transformer LLMs hold no state between calls, every call re-sends and re-encodes the full context — system prompt, tool schemas, history, retrieved memory. Part I established the quadratic resend cost; the memory-specific sharpening: the *stable* portion of an agent's context (procedural memory in the system prompt, semantic facts, tool definitions) is identical every turn, yet the model re-pays to encode it every time. On a long-horizon task the accumulated prefix is re-processed dozens of times ("the cost of intelligence compounds quadratically with task complexity" — on a 40-step task you re-send the 20K-token system prompt 40 times, and step N re-sends everything from steps 1…N−1).

Prompt caching persists the *processed state* — the KV cache — of a prefix so subsequent calls skip prefill for the matching portion. Reframed for the memory arc: **the cost of keeping something "remembered" in the context window is not fixed; it collapses by ~90% once cached.** In "Context-Window Economics" the resend cost made large in-context memory look expensive; caching makes stable in-context memory cheap to *retain* — which is why this chapter precedes "Retrieval as Memory." Whether to keep content in-context vs retrieve is dominated by whether the kept content caches well.

## How prompt caching actually works
Each token's representation is computed via attention over all preceding tokens; the intermediate key/value tensors ("KV cache") are the reusable artifact. Prompt caching stores these keyed by a hash of the token prefix. On a new request, the provider hashes the incoming prefix, checks for a match, and on a hit loads the stored KV state and runs prefill only on the novel suffix.

The prefix constraint falls out of the math: since token *k*'s state depends on tokens 1…*k*−1, cached state is valid only up to the first divergent token. Change one token anywhere in the prefix and every downstream KV entry is invalidated. Governing rule: **stable content at the front, variable content at the end.** Anthropic's docs: "Cache hits require 100% identical prompt segments, including all text and images up to and including the block marked with cache control." One character's difference is a miss.

Two mechanical details for agents:
- **Cache breakpoints / cache control.** Anthropic exposes explicit control via `cache_control: {"type": "ephemeral"}` markers (up to 4 breakpoints), caching sections that change at different frequencies. The cache key covers everything up to and including the marked block, hashed in the order **tools → system → messages**. Cascade invalidation: change a tool → busts system + messages; change system → busts messages.
- **Lookback.** Given a breakpoint, Anthropic walks backward up to 20 blocks to find the longest prefix a *prior request already wrote*. A common costly mistake: placing the breakpoint after a per-request timestamp or the user message — every request then produces a fresh write and never a read.

## Provider implementations and pricing (point-in-time, mid-2026)
The key structural distinction is **explicit vs automatic/implicit** caching, and whether there's a **write premium**.

**Anthropic (Claude) — explicit, with a write premium.** Mark cacheable blocks with `cache_control`. Multipliers on base input price: **5-minute writes = 1.25× base input; 1-hour writes = 2× base input; cache reads (hits & refreshes) = 0.1× base input (a 90% discount).** Reading refreshes the TTL free, so warm caches rarely re-pay the write. Up to 4 breakpoints; minimum cacheable prefix 1,024 tokens for Sonnet/Opus-class (some Haiku/older configs require more). Anthropic also added **automatic caching** (a single top-level `cache_control` that moves the breakpoint to the last cacheable block as the conversation grows) — the recommended default for multi-turn chat. Per-MTok figures (mid-2026):
- **Claude Opus 4.8:** $5 input / $6.25 5m-write / $10 1h-write / $0.50 cache-read / $25 output.
- **Claude Sonnet 4.6:** $3 / $3.75 / $6 / $0.30 / $15.
- **Claude Haiku 4.5:** $1 / $1.25 / $2 / $0.10 / $5.
- **Claude Sonnet 5:** $2 / $2.50 / $4 / $0.20 / $10 (introductory through Aug 31 2026; rises to $3/$15 base after).
Caches are isolated between organizations, and (as of Feb 5 2026 on the Claude API/AWS/Foundry) per-workspace within an organization. ZDR-eligible; only KV representations and hashes held in memory, never raw prompt text at rest.

**OpenAI — automatic, no write fee.** Fully automatic (no code, no markers) on all recent models. Engages on prompts ≥1,024 tokens, matching the longest previously-computed prefix in 128-token increments. **Current cached-input discount is 90% — cached input billed at 10% of standard input** (per OpenAI's official pricing page, mid-2026): GPT-5.5 is $5.00/M input and **$0.50/M cached** ($30/M output); GPT-5.4 is $2.50 / $0.25 / $15. The often-cited "50%" is the *original* Oct 2024 launch discount, now outdated; "75%" applied to GPT-4.1. **No cache-write fee** — caching "has no additional fees associated with it." (Reports of a "1.25× write premium for GPT-5.6+" conflate Anthropic's model; that is not OpenAI's pricing.) Retention: a `prompt_cache_retention` parameter with `in_memory` (classic 5–10 min inactivity, up to 1 hr) and `"24h"` (extended, KV offloaded to GPU-local storage). **For GPT-5.5+, `24h` is the default and only option;** older models support both. Retention length doesn't change the per-token rate. Routing by a hash of the first ~256 tokens; optional `prompt_cache_key` improves hit-rate; exceeding ~15 req/min for one prefix+key can overflow and reduce effectiveness. Best-effort — one test saw a warm ~3,700-token prompt hit ~89% from cache but still register occasional mid-streak misses, and a ~1,300-token prompt never cache despite exceeding the nominal minimum.

**Google Gemini — both implicit and explicit.** *Implicit* caching on by default for Gemini 2.5+, passing through a **90% discount (10% of input) on 2.5+ (75% on 2.0)** automatically, no storage fee, no guarantee. *Explicit* context caching (`cachedContents` API) gives a *guaranteed* discount plus TTL control (default 60 min, updatable/deletable) but adds an **hourly storage fee** based on cached-token count and TTL. Minimums higher for explicit (a 2,048-token floor historically; some configs cite 32,768) vs ~1,024 (Flash) / 2,048 (Pro) for implicit. Worked example (Google-reported, June 9 2026): a 100K-token system prompt re-sent across 1,000 requests on Gemini 3.1 Pro drops from ~$200 naive input to ~$20.50 with explicit caching (write + reads + ~1 hr storage), ~90%. On cheap Flash-tier the hourly storage fee can *exceed* per-read savings, so implicit is the rational default there; reserve explicit for Pro-tier and genuinely high-reuse Flash paths. Explicit supports VPC Service Controls.

**DeepSeek — automatic disk-based, cheapest absolute prices.** Pioneered large-scale **hard-disk KV caching**, enabled by MLA (Multi-head Latent Attention) compressing the KV cache enough to store cheaply on disk. Automatic and free to store, with isolation per user. [DeepSeek's Models & Pricing page](https://api-docs.deepseek.com/quick_start/pricing/) lists V4 Flash input at $0.14/M for cache misses and $0.0028/M for cache hits; it lists V4 Pro at $0.435/M miss and $0.003625/M hit. Prices change, so recheck that page before budgeting. DeepSeek's original announcement reported first-token latency for extreme 128K-input repeat scenarios dropping from ~13 s to ~500 ms. Best-effort disk caching: lifetime "a few hours to a few days," no SLA; track `prompt_cache_hit_tokens` / `prompt_cache_miss_tokens`.

**AWS Bedrock & Azure/Microsoft Foundry.** Bedrock offers prompt caching (GA with Claude and Amazon Nova) reporting up to 85% latency and 90% cost reductions; Bedrock does **not** support Anthropic's automatic caching (explicit breakpoints only) and uses org-level isolation. Azure OpenAI (Foundry) mirrors OpenAI: automatic, ≥1,024-token minimum, 128-token increments, cached tokens discounted on Standard deployments and up to 100% discount on Provisioned; no opt-out.

**Comparative picture.** Anthropic and OpenAI match on the 90% read discount; the difference is operational. OpenAI wins on simplicity (automatic, free writes, 24h retention). Anthropic wins on control (breakpoints for mixed-frequency content, explicit 1-hour TTL for spaced traffic) at the cost of a write premium and marker discipline. Gemini is most flexible (implicit + explicit) but explicit carries storage cost. DeepSeek is cheapest in absolute dollars and zero-config, but best-effort with no TTL guarantees.

## The economics — cost and latency
**Cost.** For a large stable prefix read many times before TTL expiry, cached reads at 10% of input push the input-token bill toward one-tenth of uncached cost. Break-even is governed by any write premium. On Anthropic's 5-min TTL, two hits average (1.25 + 0.10)/2 = 0.675× (32.5% saving on the cached portion); three hits average 0.483× (52%); the asymptote as the cache stays warm is 0.10×. **On the 5-min TTL, break-even arrives at the second hit.** OpenAI, with free writes, has no write break-even — the discount applies from the first hit. Worked mid-conversation example (system + tools = 6,000 cached tokens, 2,000 variable tokens/turn, 8 turns): ~59% reduction in billable input-token equivalents.

Real-world realized figures:
- **ProjectDiscovery** (Neo agentic security platform, 20–40+ LLM steps against a 20,000-token system prompt): overall LLM cost savings of **59%**, peaking at **70%** over the final 10 days, by lifting cache hit rate from single digits to **84%** via explicit breakpoints and 1-hour TTLs. Their single biggest win — moving dynamic content out of the cacheable prefix — took hit rate from **7% to 74%** in one deployment; they ultimately served **9.8 billion tokens from cache**.
- In *"Don't Break the Cache: An Evaluation of Prompt Caching for Long-Horizon Agentic Tasks"* (Lumer et al., arXiv 2601.06007), the authors evaluate OpenAI, Anthropic, and Google on DeepResearch Bench with 10K-token system prompts over 500+ sessions: prompt caching **reduces API costs by 41–80% and improves TTFT by 13–31% across providers.** Cost savings scaled with *prompt size* (10–45% at 500 tokens to 54–89% at 50,000 tokens) far more than with tool count — cacheable prefix length (dominated by the system prompt) is the key lever.

**Latency.** Cache hits skip prefill, so TTFT drops sharply. Anthropic reports reductions **up to ~85% for long prompts** (a 100K-token document going from ~11.5 s to ~2.4 s). OpenAI reports **up to ~80%** TTFT reduction; one cookbook test found cached requests 7% faster at 1,024 tokens but **67% faster at 150K+ tokens** — the benefit grows with prefix length. The mechanism turns per-request latency from roughly proportional to *total context length* into roughly proportional to *generated output length*.

**When caching does NOT help:** constantly-changing prefixes (timestamps, per-user salts, freshly-retrieved bundles at the front bust the cache every request — you pay writes, never read); one-shot / very low-volume calls (on write-premium models, requests spaced beyond TTL make the premium pure overhead; OpenAI's free writes avoid this); content below the ~1,024-token minimum; output-bound workloads (a 50-token prompt generating 5,000 output tokens is unaffected — caching only touches input). Critically, **naive full-context caching in agents can *increase* latency** — the "Don't Break the Cache" paper found placing dynamic content at the end and excluding dynamic tool results "provides more consistent benefits than naive full-context caching, which can paradoxically increase latency."

## Cache-aware prompt and memory design
The golden rule maps onto the memory taxonomy:
- **Front (caches well):** procedural memory (system prompt, instructions, policies), tool definitions, few-shot examples, stable semantic memory (reference documents, long-lived facts) — byte-stable across turns.
- **End (never/rarely cached):** the latest user message, freshly-retrieved episodic memory, dynamic state, timestamps.

The tension with dynamic retrieval is fundamental: **injecting a freshly-retrieved memory into the middle invalidates the cache for everything after it.** Retrieved/dynamic content must go near the end, after the cached prefix. This is the concrete mechanism by which caching and retrieval interact — and why the chapter ordering (caching, then retrieval) is deliberate: you can't design a retrieval-injection strategy without first understanding the cache boundary it must respect.

For conversation history, the append-only pattern keeps the growing prefix cache-consistent: each turn adds to the end, so the prefix through turn N−1 stays identical and cacheable on turn N. Anthropic's automatic caching implements exactly this, sliding the breakpoint forward each turn. A production 4-breakpoint layout for a long agentic session (one breakpoint on tools+system with 1-hour TTL that never moves, plus a sliding pair over recent turns) has reduced a 25-turn investigation to ~20% of uncached input cost.

## Caching and agentic workloads specifically
Agents are the canonical fit. An agent loop makes many sequential calls, each re-sending system prompt + tools + a monotonically-growing history. The stable head (system + tools) is identical every call, and each turn appends a tool call + result to the tail — so everything before the new tail stays cached. Anthropic's guidance: caching is "not optional for agentic systems"; if agents run more than 3–5 steps, uncached operation leaves substantial money on the table, and multi-step tasks are simultaneously the most expensive *and* the most cacheable.

The interaction with **context editing / memory management** is a genuine tradeoff. Anthropic's server-side context editing (beta header `context-management-2025-06-27`) clears stale tool results and thinking blocks *at the API level, after the prompt-cache lookup* — so it can prune context **without destroying the cached prefix**, a deliberate advantage over client-side stripping (which would change the prefix and bust the cache). Anthropic reports context editing alone yields a 29% performance improvement, and combined with the file-based memory tool, 39%, with an 84% token reduction on a 100-turn web-search evaluation (self-reported internal benchmarks — validate on your own workloads). General principle: **any client-side edit to already-cached context trades context hygiene against cache retention.** Clearing an old tool result to save tokens can cost a full re-prefill of everything downstream. Server-side editing that runs after cache lookup gets both.

For **multi-agent** systems, each agent's stable prefix (its own system prompt + tools) caches independently; a shared system prompt across a swarm can be cached once and reused across all agents (where self-hosted RadixAttention shines — below).

## Advanced topics and the frontier
**Self-hosted KV cache management.** The provider APIs abstract raw KV-cache management self-hosters do directly:
- **vLLM / PagedAttention** (Kwon et al., SOSP 2023): manages the KV cache like OS virtual memory — fixed-size *blocks* in non-contiguous physical memory, eliminating fragmentation. Automatic prefix caching maps logical blocks to a global hash table so blocks sharing a hash (e.g., a shared system prompt) map to the same physical block and are reused across requests; reference-count-aware LRU eviction.
- **SGLang / RadixAttention** (Zheng et al., NeurIPS 2024, arXiv 2312.07104): organizes the KV cache as a *radix tree* indexed by token sequences, enabling automatic runtime detection of the longest shared prefix and multi-level sharing (few-shot examples, branching, chat history, agent templates). LRU eviction plus cache-aware scheduling; reports **up to 6.4× higher throughput** vs prior systems on agent control, few-shot, JSON decoding, RAG, and multi-turn chat — degrading sharply if prefix ordering is inconsistent. For multi-agent swarms sharing a system prompt, near-zero-cost memory sharing and graceful fork handling. (RadixAttention finds the *longest matching prefix*, enabling partial hits, whereas provider prompt caching is binary — the whole marked prefix matches or it doesn't.)
- **KV cache offloading / LMCache:** extends vLLM's in-GPU prefix cache into a tiered hierarchy (GPU HBM → CPU DRAM → NVMe SSD → remote/object storage), reusing KV across requests, sessions, even engine instances. Its documented local-disk example: cold TTFT 6.3 s → warm TTFT 0.148 s. How self-hosters achieve provider-like "persistent" caching beyond GPU memory limits.

**Cache-Augmented Generation (CAG).** *"Don't Do RAG: When Cache-Augmented Generation is All You Need for Knowledge Tasks"* (Chan, Chen, Cheng, Huang; arXiv 2412.15605, ACM Web Conference 2025) preloads an *entire* (small, static) knowledge base into extended context, precomputes/caches its KV state, and answers queries directly against the cache — eliminating retrieval. Tradeoffs: removes retrieval latency and retrieval-selection errors, simplifies architecture, but requires the whole corpus to fit in context, and performance can degrade at very long context lengths. Best when the knowledge base is domain-specific, relatively static, and small enough — the direct counterpoint to the retrieval chapter: small and stable → cache it all; large or churning → retrieve.

**Semantic caching (distinct concept).** Prompt caching reuses *processed input tokens* on exact-prefix matches. **Semantic caching** operates at the application layer: embed the incoming query, do vector-similarity search against stored query→response pairs, and if similarity exceeds a threshold, return the stored *response* without calling the model. Differences: semantic caching bypasses the LLM entirely (saving input *and* output cost, collapsing latency to ms) but risks returning a subtly-wrong answer when two queries embed as similar but differ in meaning; prompt caching never changes the output. Mature stacks layer them: exact-match cache → semantic cache (catches paraphrases) → prompt cache (discounts novel queries that still hit the model). Redis LangCache and GPTCache are representative; the similarity threshold (~0.85–0.87) is the key correctness knob.

**Security / privacy.** Shared KV caches create a **timing side-channel.** Because cache hits return faster (lower TTFT) than misses, an attacker can probe with crafted prefixes and measure latency to infer whether a prefix was recently cached — and, token-by-token, reconstruct another user's prompt. A Stanford audit (Gu, Li, Kuditipudi, Liang, Hashimoto; arXiv 2502.07776, ICML 2025) of 17 commercial APIs (Sept–early Oct 2024) "detected prompt caching in 8 out of 17 API providers, and… detected global cache sharing in 7 providers" — cache hits averaging ~0.1 s vs ~0.5 s for misses, a measurable gap enabling cross-user inference. Follow-on: "The Early Bird Catches the Leak" (Zheng et al., arXiv 2409.20002; InputSnatch/PromptPeek), NVIDIA's KV-cache hardening guidance, and defenses in *PrefixWall* (arXiv 2603.10726) and *Selective KV-Cache Sharing* (arXiv 2508.08438). Mitigations: per-organization (and now per-workspace, on Anthropic) cache isolation, which the major providers implement — Anthropic, OpenAI, and Gemini all state caches are never shared across organizations. Residual risk is within-org and at gateway layers, plus unresolved GDPR/data-residency questions about where KV tensors physically live. For regulated data, self-hosted inference with private caches remains conservative.

## Practical guidance and synthesis
**Decision framework for repeatedly-needed information.** Choose among three levers:
1. **Keep it in cached context** when it is (a) stable/byte-identical across calls, (b) small enough to sit in the window without crowding out working memory, and (c) reused frequently enough to stay warm within the TTL. Best for system prompts, tool definitions, few-shot examples, and small static reference corpora (the CAG case).
2. **Retrieve it on demand** when the corpus is large, changes often, or is only occasionally relevant — inject retrieved chunks at the *end* so they don't bust the cached prefix (the next chapter).
3. **Bake it into the model** (fine-tuning) when the knowledge is truly static, high-value, and reused across *all* traffic — amortized over enough calls, weights beat both context and retrieval, at the cost of update latency and training complexity.

**Operational rules:** order stable→volatile, always (captures most of the value on every provider); choose explicit vs automatic by provider/need (default to automatic for simplicity; reach for explicit breakpoints when you have mixed-frequency sections or need guaranteed savings/TTL control — only when you can measure reuse); keep the prefix above ~1,024 tokens or caching won't engage; use the 1-hour TTL for shared, spaced-out content (Anthropic) and warm the cache with a few dummy requests at deploy time; never cache dynamic tool results or timestamps in the prefix (caching them can *raise* latency); instrument relentlessly (log `cache_read_input_tokens`/`cache_creation_input_tokens` on Anthropic, `cached_tokens` on OpenAI, `cachedContentTokenCount` on Gemini, `prompt_cache_hit_tokens` on DeepSeek; aim for >60% hit rate on long-context paths, 80%+ achievable); treat pricing as point-in-time (build price variables into cost estimators; re-verify before committing budgets).

**The synthesis — the economics of remembering.** Caching, retrieval, and recompute are three prices for the same good: having the right information in front of the model at inference time. Caching makes *keeping* cheap (≈10% of input cost for stable, warm content); retrieval makes *recall* cheap (pay only for what you fetch, when you fetch it); recompute/summarization makes *forgetting-and-reconstructing* cheap when reuse is low. The competent architect places stable procedural and semantic memory in a cached front, injects dynamic episodic memory via retrieval at the end, prunes with server-side (cache-preserving) context editing, and reserves fine-tuning for the truly universal. The cache boundary is the seam where these levers meet, and designing that seam well is what makes long-running agents economically viable.

## Caveats
- All pricing, TTLs, minimums, and model names are point-in-time (mid-2026) and change fast; re-verify against official pricing pages. Model lineups cited (Claude Opus 4.8 / Sonnet 5, GPT-5.5/5.4, Gemini 3.1 Pro, DeepSeek V4) are mid-2026 snapshots.
- Several headline benefit figures are self-reported: Anthropic's 85% latency / 90% cost figures and the 29%/39% context-editing gains are vendor benchmarks; the ProjectDiscovery 59–70% figures are one team's production numbers; the 41–80% cost / 13–31% TTFT figures come from a single (peer-reviewed) academic evaluation on one benchmark. Directionally consistent, but validate on your own workloads.
- Caching is best-effort on automatic-caching providers (OpenAI, DeepSeek): no hit guarantee; routing, load, and prefix length all affect whether a call hits; real-world hit rates fall below the theoretical maximum, with occasional misses even on warm prefixes.
- The 90% "discount" applies only to the cached input tokens, not the whole bill — the variable tail is full price, output tokens entirely unaffected. Effective savings depend on the ratio of stable prefix to variable tail and on output volume.
- Genuine debates: whether long-context CAG can truly replace RAG (context-rot and long-context degradation cut against it); the right similarity threshold for semantic caching (a correctness/coverage tradeoff with no universal answer); and how to reconcile aggressive context editing (helps focus and cost) with cache retention (aggressive client-side editing undermines it).
