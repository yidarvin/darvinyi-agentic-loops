# Self-Managed Memory

Research reference for *Agentic Loops*, Chapter 18 (fourth and final chapter of Part IV, Memory; the capstone of the memory arc). Follows "The Memory Taxonomy," "Prompt Caching and the Economics of Remembering," and "Retrieval as Memory" (the READ side). Current as of 2026. The reader already knows the taxonomy, the lifecycle, caching, and retrieval mechanics; this chapter is the deep treatment of the WRITE side and specifically the agent managing its OWN memory. It also synthesizes the whole arc. Fast-moving area; several vendor benchmark numbers are self-reported or contested — version-pin and re-verify.

## TL;DR
- **Self-managed memory is the write side of the read-write cycle put in the agent's own hands**: rather than a fixed developer pipeline extracting and storing memories, the agent decides — via tool calls and reasoning — what to remember, how to encode it, and when to update or forget. As of 2026 this is a real, shipping capability: Letta's editable memory blocks and self-editing functions, Anthropic's client-side file-based memory tool, LangMem's hot-path tools, and Mem0's LLM-routed ADD/UPDATE/DELETE all instantiate it.
- **The frontier technique is offloading memory work out of the interaction loop.** Sleep-time compute (Lin et al., arXiv:2504.13171) and background "dream"/sleep-time agents reorganize and consolidate memory during idle time, cutting test-time compute ~5× on stateful reasoning tasks and improving memory quality without adding user-facing latency. Reflection (Generative Agents) and consolidation (episodic→semantic) are the higher-order operations that make accumulated memory useful rather than merely large.
- **Self-management's power is also its danger.** Agents write low-quality or wrong memories that propagate errors; memory stores are an attack surface (MINJA query-only poisoning exceeds 95% injection success); and write-side quality is the dominant failure mode (HaluMem shows errors originate and amplify at the extraction/update stage). The pragmatic 2026 posture is scoped autonomy: agent proposes, system validates, with hard limits, provenance, and direct write-side evaluation.

## The write side and the shift from infrastructure to capability
The reader knows the lifecycle (write/store/retrieve/forget) and the taxonomy; the previous chapter treated retrieval as the READ side. This chapter is the WRITE side, and specifically the agent performing it *itself*.

The industry uses "agent memory" for two quite different architectures:
- **Externally-managed memory.** A developer-defined pipeline outside the agent's reasoning loop. After each turn (or in a background job), an extraction model reads the conversation, decides what's worth keeping, and writes it; retrieval later pulls it back. Mem0 is the archetype: an extraction phase distills salient facts from each message pair, then an LLM-based routing controller inspects the top-k most similar existing memories and classifies each candidate as ADD, UPDATE, DELETE, or NOOP. The agent never "decides" to remember; the memory system does. Memory as *infrastructure*.
- **Self-managed memory.** The agent itself is given memory as tools (or an editable region of its own context) and decides, as part of normal reasoning, what to write and when. Letta's framing is sharpest: "the agent has a memory and the model has tools to edit it. Forgetting to remember requires actively not calling the tools." Memory as an *agent capability*.

Why self-management matters: it lets the agent adapt its memory policy to context in ways a fixed pipeline cannot. A static extractor applies the same heuristic to every turn; a self-managing agent can decide *this* offhand remark ("I'm lactose intolerant") is worth persisting while *that* verbose tool output is not, using the same situated judgment it applies to the task. It's the endpoint of the shift the whole book traces: from stateless functions to persistent processes that learn. The cost is equally real: the agent's judgment about what to remember is now load-bearing, and models are not yet reliably good at it.

## Self-editing memory: MemGPT / Letta in depth
The foundational mechanism is **self-editing memory via tool calls**, from "MemGPT: Towards LLMs as Operating Systems" (Packer, Fang, Patil, Lin, Wooders, Gonzalez; 2023). Central analogy: treat the context window like a computer's physical RAM and external stores like disk, and give the model tools to page information between them — "virtual context management" providing the illusion of unbounded context. MemGPT became **Letta** in September 2024 (Berkeley Sky Computing Lab).

**Memory blocks.** The load-bearing primitive: a labeled, bounded (character-limited) section of the context window the agent can edit and that persists across sessions. Conventional blocks are `human` (what the agent knows about the user) and `persona` (self-description); custom blocks (`project`, `current_work`, `lessons_learned`) are common. Blocks render directly into the system prompt every turn, so they're always "hot" — no retrieval call. One production coding-agent setup kept blocks to ~2,000 characters each, with a policy in the persona block to offload detail to archival storage on overflow.

**The memory hierarchy** (the MemGPT OS analogy, three tiers):
- **Core memory** — the editable blocks, always in-context (RAM).
- **Recall memory** — conversation history, searchable on demand (`conversation_search`), evicted under context pressure but never lost (disk cache).
- **Archival memory** — a long-term external vector store the agent queries with `archival_memory_search` (cold storage).

**The self-editing functions.** The original MemGPT agent ships base tools: `send_message`, `core_memory_append` and `core_memory_replace` (edit in-context blocks), `conversation_search` (recall), archival insert/search. The critical test is contradiction handling: when the user says "my favorite color is actually blue, not red," a well-behaved agent emits `core_memory_replace(...)` rather than appending a conflicting fact. Letta guards this with read-only block protection, line-number validation, and a uniqueness check that `old_str` appears exactly once before a replace. When context exceeds a threshold, older messages are evicted to recall and `partial_evict_summarization()` compresses them.

**How the agent decides to write.** Because reasoning is exposed (an inner-monologue field, or native reasoning tokens), the write decision is legible: the agent reasons "I need to update the human block," then calls the tool. Letta's newer v1 agent architecture (for frontier reasoning models) drops the heartbeat mechanism and split thinking/speaking in favor of native loops.

**Letta as of 2026.** **Shared memory blocks** can be accessed by multiple agents, enabling coordination through common state. **Sleep-time agents** (below) run in the background against shared blocks. The platform ships an Agent Development Environment (ADE) for inspecting memory state across tiers, and **Letta Code** (a memory-first coding agent, March 2026) stores memory in **MemFS**, a git-backed "context repository" that projects memory blocks into markdown files with version history and conflict resolution; CLI verbs include `/remember`, `/sleeptime`, `/doctor` (audit memory quality), `/palace` (view memory), plus a memory-defragmentation flow that splits large files, merges duplicates, and restructures the hierarchy.

A characteristic failure mode: because the agent spends part of its reasoning budget on memory housekeeping, it can learn that "calling `core_memory_append` makes the loop feel productive" and substitute memory edits for actually answering — "the output gets thin while the memory blocks grow." Fix: separate in the system prompt "what you do for the user" from "what you do to memory," and require the user-facing answer first.

## Claude's memory tool and file-based self-managed memory
Anthropic's **memory tool** (public beta September 29 2025, all Claude 4+ models) is a different substrate for the same idea: the filesystem as memory. Its entire configuration is `{"type": "memory_20250818", "name": "memory"}` — no input schema. Claude issues memory operations as tool calls; the developer's application executes them. Six client-side commands over a `/memories` directory: `view` (list a directory or read a file, optional line ranges), `create`, `str_replace`, `insert` (at a line), `delete`, `rename`.

**The client-side execution model** is the key design choice. Because the tool is client-side, Claude only *requests* operations; the developer's handler executes them against storage they control — a per-user directory, a database, or a service (Supermemory maps `/memories/x.txt` onto documents). Memory "lives entirely in your application." A later conversation continues from the same memory simply by sending the same tools entry against the same store. Anthropic auto-injects a system prompt (~2,500 tokens) instructing Claude to check memory first: "IMPORTANT: ALWAYS VIEW YOUR MEMORY DIRECTORY BEFORE DOING ANYTHING ELSE." The filesystem metaphor is deliberate — Claude is heavily trained on file operations, so little extra instruction is needed.

**Combination with context editing.** The memory tool pairs with **context editing** (beta header `context-management-2025-06-27`), which automatically clears stale tool calls/results from the window as it approaches a threshold — server-side, after prompt-cache lookup, so it doesn't destroy cache prefixes. Division of labor: context editing clears working memory while memory files preserve what matters. Per Anthropic's internal evaluations: "combining the memory tool with context editing improved performance by 39% over baseline. Context editing alone delivered a 29% improvement. In a 100-turn web search evaluation, context editing… reducing token consumption by 84%." Self-reported internal numbers, not independent benchmarks — validate on your own workloads.

**Filesystem-as-memory lineage.** Descends from MemGPT's OS roots and **Manus's** production practice of treating the file system as "unlimited external context" with "recoverable/restorable compression" — dropping a web page's content as long as the URL is preserved, or a document's body as long as its path remains. Manus also has the agent continually rewrite a `todo.md` to recite goals into the end of context. The pattern generalizes: a small set of filesystem tools (plus `glob`/`grep`) lets an agent use a computer as memory without a vector store.

**Agent Skills and CLAUDE.md as self-managed procedural memory.** Anthropic's **Agent Skills** (open standard December 18 2025; adopted by OpenAI Codex, Gemini CLI, GitHub Copilot, Cursor) are folders containing a `SKILL.md` (YAML frontmatter with `name`/`description` plus a markdown body) and optional bundled scripts/reference files, implementing **progressive disclosure**: at startup only each skill's name+description (~80 tokens median) load; the full body (~2,000 tokens median) loads only when the agent judges the skill relevant; deeper reference files load only when a step needs them. In taxonomy terms, Skills are **self-managed procedural memory** — packaged "how-to" knowledge the agent discovers and loads on demand — and Letta Code's "skill learning" and Claude Code's `CLAUDE.md` persistent instructions are the same idea. Claude Code's newer auto-memory (v2.1.59, February 26 2026) has the agent maintain and even reorganize project memory files itself.

## Reflection, consolidation, and memory maintenance
Simple writes produce a log; higher-order operations produce a *memory*.

**Reflection.** The canonical mechanism is from **Generative Agents: Interactive Simulacra of Human Behavior** (Park et al., UIST 2023). Agents store experiences in an append-only natural-language *memory stream* (each entry timestamped, with an LLM-assigned importance score 1–10). Retrieval scores by a weighted sum of **recency** (exponential decay), **importance** (the stored score), and **relevance** (embedding similarity), roughly equally weighted. Because raw observations don't let agents generalize, the architecture adds **reflection**: periodically — triggered when the sum of recent importance scores crosses a threshold, in practice ~2–3 times a "day" — the agent generates salient questions over recent memories and synthesizes higher-level insights ("Klaus has been eating alone and seems withdrawn"), written back into the stream and retrieved like any other memory. The paper's lesson: believable long-horizon behavior depends more on this external memory-and-reflection architecture than on the base model. Reflexion (Shinn et al., NeurIPS 2023, arXiv:2303.11366) is a narrower cousin — post-failure self-critique prepended to the next attempt; "Reflexion achieves a 91% pass@1 accuracy on the HumanEval coding benchmark, surpassing the previous state-of-the-art GPT-4 that achieves 80%."

**Consolidation.** Converts *episodic* memories (specific events) into *semantic* memories (durable general facts), summarizes conversations into lasting memories, merges related memories, moves working memory to long-term storage. LangMem frames this as balancing over-extraction (hurts precision at search time) against under-extraction (hurts recall), with a tunable "memory enrichment" process. Consolidation keeps a store useful as it grows: without it, notes "accumulate the same way unconsolidated short-term memories do — contradictions persist, outdated entries linger, and the signal-to-noise ratio degrades with every session."

**Maintenance.** Updating memories when facts change, resolving contradictions, deduplication, forgetting/pruning stale entries. Claude Code's **Auto Dream** is a shipped example: a four-phase consolidation that converts relative dates to absolute ("yesterday we decided to use Redis" → "On 2026-03-15…"), deletes contradicted facts (switched Express→Fastify → drop "API uses Express"), removes stale notes, and merges overlapping entries — in a sandboxed, read-only-to-code, lock-file-guarded background process. A genuine design split: Mem0's *newer* algorithm deliberately moved to **ADD-only** extraction (preserving both old and new facts with temporal context rather than overwriting); per Mem0's official GitHub it scores "91.6 on LoCoMo — +20 points over the previous algorithm; 94.8 on LongMemEval — +27 points, with +53.6 on assistant memory recall," pushing conflict resolution onto retrieval-time temporal ranking. This is the opposite of the `core_memory_replace` philosophy — evidence that "should the write step overwrite or accumulate?" is genuinely unsettled.

**Learning from experience.** The most ambitious write is procedural: agents that write lessons and procedures to memory and improve over time. LangMem models three memory types to learn (semantic facts, episodic experiences stored as few-shot examples, procedural rules refined via prompt-optimization algorithms — `metaprompt`, `gradient`, `prompt_memory`). ACE (below) and Letta Code's "system prompt learning through memory blocks" share the instinct: accumulated context becomes a growing skill library.

## Sleep-time compute and background memory processing
The defining 2026 pattern: doing memory work *outside* the interaction loop.

**Sleep-time compute.** "Sleep-time Compute: Beyond Inference Scaling at Test-time" (Lin, Snell, Wang, Packer, Wooders, Stoica, Gonzalez; arXiv:2504.13171, April 2025). The idea: instead of a stateless model that only "thinks" when a query arrives (test-time compute), let it think offline about its context before queries come — "anticipating what queries users might ask and pre-computing useful quantities." Per the paper, "sleep-time compute can reduce the amount of test-time compute needed to achieve the same accuracy by ~5x on Stateful GSM-Symbolic and Stateful AIME and… by scaling sleep-time compute we can further increase accuracy by up to 13% on Stateful GSM-Symbolic and 18% on Stateful AIME"; the authors also report ~2.5× lower cost per query in a multi-query setting. Reasoning during sleep transforms "raw context" into "learned context" — explicitly analogous to human sleep and memory consolidation.

**Background/sleep-time agents.** Letta operationalizes this as a dual-agent architecture: set `enable_sleeptime: true` and Letta creates a primary agent (conversation, with `conversation_search`/`archival_memory_search`) plus a sleep-time agent whose tools *manage the primary agent's memory blocks*. The sleep-time agent reflects on raw context (conversation history, uploaded files) to iteratively derive "learned context," writing it into shared memory blocks. A specialized case of the multi-agent pattern: memory management as a dedicated background role. In Letta Code it's "dreaming" (`/sleeptime`); Claude Code's Auto Dream is a parallel implementation.

**Economics and latency.** The core benefit: memory work adds no latency to the user-facing turn, and can run on cheaper/batched compute during idle time. The primary agent "stays lightweight and deterministic under user load," while the sleep-time agent "can think deeply, call specialized tools, and update memory blocks." This fixes a known MemGPT weakness (bundling memory management into the single interaction agent slowed responses; splitting improves both response time and memory quality). The tradeoff: sleep-time work is speculative — you spend compute anticipating queries that may not come — so it pays off most for stable, reused context.

## Challenges and failure modes
Self-management is where memory becomes dangerous. The honest state of the art:

**Deciding what's worth remembering (signal vs noise).** The central open problem. Remember too much → pollute future context and bloat the store; too little → lose the point of persistence. Letta's own guidance — "NG: 'Delete the oldest item if it exceeds 5 items'; OK: 'If Core Memory seems to be overflowing, move old information to Archival'" — is a tacit admission that precise policies are brittle and vague policies delegate judgment to a model that may not have it.

**Error propagation through memory.** A wrong memory, once written, persists and corrupts every future interaction that retrieves it — self-reinforcing mistakes. HaluMem (Chen, Niu, Li, Liu, Zheng, Tang, Li, Xiong, Li; arXiv:2511.03506, Nov 2025 / rev. Jan 2026), the first *operation-level* memory hallucination benchmark, quantifies this: existing memory systems "tend to generate and accumulate hallucinations during the extraction and updating stages, which subsequently propagate errors to the question answering stage," and "these upstream hallucinations are often amplified during the generation stage." Concretely, across systems (Mem0, Mem0-Graph, Memobase, MemOS, Supermemory, Zep) the write stage fails hardest: on HaluMem-Medium, extraction recall ranges ~74% (best, MemOS) down to ~15% (Memobase), and at 1M-token scale (HaluMem-Long) some collapse to single-digit recall (Mem0 ~3%). Memory-*updating* is dominated by **omission** — best-case ~37% omission (MemOS), typically 74–98% — meaning systems overwhelmingly *fail to write* a needed update rather than fabricating one; update hallucination is under ~1.2%. Downstream, QA hallucination on Medium ranges ~15% (MemOS) to ~30% (Memobase). The lesson: the write side, not retrieval, is the primary error source, dominated by omission and low extraction accuracy.

**Contradiction and staleness.** Knowing when a memory is outdated is itself a hard judgment; the STALE line of work (2026) explicitly asks whether agents can tell when their memories are no longer valid. The temporal-knowledge-graph approach (Zep/Graphiti) addresses this by storing validity windows and invalidating-but-preserving superseded facts rather than deleting them.

**Memory poisoning / security.** Persistent self-managed memory is an attack surface. **MINJA** — "A Practical Memory Injection Attack against LLM Agents" (Dong, He, Tang, Liu; arXiv:2503.03704) — shows an attacker can poison an agent's memory bank through *query-only* interaction (no privileged access), using bridging steps, an indication prompt, and progressive shortening. Per the paper, "MINJA achieves over 95% ISR [injection success rate] across all LLM-based agents and datasets, and over 70% ASR [attack success rate] on most datasets"; overall ISR "uniformly exceeds 95% with a low variance, while the overall ASR ranges from around 60% to more than 90%," surpassing 90% for a GPT-4-based EHR agent on eICU and a GPT-4o-based agent on Webshop. Because the injected records are semantically plausible task-aligned reasoning, they evade input/output moderation (e.g., Llama Guard) and memory sanitization. Follow-on work extends this to healthcare/EHR agents (arXiv:2601.05504) and to "contagious"/worm-style propagation through *shared* memory (AgentPoison and cross-platform worm studies) — especially concerning for shared-memory-block architectures (Letta, multi-agent). Defenses: memory isolation, strong user authentication, provenance/typed memory with runtime-mediated promotion (separating untrusted candidate memory from trusted memory), privacy filtering. Privacy is first-class: Anthropic notes Claude "will usually refuse to write down sensitive information," and recommends per-user root directories, size caps, and consent filters, since persistent memory brings GDPR-type obligations.

**Reliability of self-assessment.** Agents may not know what they'll need later, may misjudge importance, may write low-quality memories — the HaluMem omission numbers are direct evidence. RealMem (arXiv:2601.06966, Jan 2026) shows that in realistic long-term project scenarios, state-of-the-art memory systems "struggle to maintain coherent project threads," with a "substantial gap between all evaluated methods and the Oracle upper bound."

**Evaluation.** The taxonomy chapter covered LoCoMo (Maharana et al., 2024 — ~10 multi-session dialogues, ~600 turns) and LongMemEval (Wu et al., 2025 — 500 questions over five abilities: information extraction, multi-session reasoning, temporal reasoning, knowledge updates, abstention). Those are largely *read-side* end-to-end benchmarks. The write-side-specific frontier is newer: HaluMem (operation-level extraction/update/QA metrics), RealMem (proactive project-memory use), and a cluster of 2026 studies on error attribution and management quality (MemTrace on error tracing, MemSyco-Bench on memory-induced sycophancy, MemoryRewardBench/Fine-Mem on reward models for memory management). Practical takeaway: end-to-end QA accuracy *hides* upstream memory errors — evaluate the write step directly.

## Architectural patterns and the frontier (2026)
**The autonomy spectrum.** Systems range from fully-manual (developer controls all memory) to fully-autonomous (agent controls all memory), with a pragmatic middle where the agent proposes and the system validates, and self-management is *scoped*. The evidence argues strongly for the middle: unconstrained self-management produces the HaluMem/MINJA failure modes, while fully-manual pipelines can't personalize or adapt. Scoping means hard size caps, path/namespace isolation, provenance tags, read-only protection on trusted blocks, and a human- or rule-validated promotion step from candidate to durable memory.

**Memory-as-tools vs memory-as-a-service.** Two philosophies: the agent calls memory *tools* it reasons about (Letta blocks, Claude memory tool, LangMem's `create_manage_memory_tool`/`create_search_memory_tool` "hot path"), versus a framework/service that handles memory *for* the agent (Mem0's extraction pipeline, LangMem's background manager, Zep). LangMem offers both modes in one SDK, and its guidance — decide *what* to learn, *when/who* forms memories, and *where* they're stored/recalled — is a good checklist. Production caveat: hot-path self-editing adds tokens and latency to every turn. Per the Mem0 paper (Chhikara et al., arXiv:2504.19413), "LangMem exhibits even higher search latencies (p50: 17.99s, p95: 59.82s), rendering it impractical for interactive applications," versus Mem0's ~0.200s p95 — measure latency before shipping any hot-path configuration.

**Integration with the stack.** Self-managed memory increasingly fuses with the rest of the system: filesystem/code-execution tools as substrate (Manus, Skills, MemFS), retrieval as the read side over archival memory, multi-agent architectures with shared memory blocks and dedicated memory agents, and skills as portable procedural memory. The MemFS git-backed model — memory as a versioned, diffable, syncable repository — is a notable convergence of memory with software-engineering tooling.

**Emerging frameworks and standards.** As of 2026: Letta (self-editing blocks, sleep-time agents, MemFS), Mem0 (LLM-routed extraction, now ADD-only), LangMem (hot-path + background, three memory types), Claude's memory tool + context editing, A-MEM (arXiv:2502.12110, NeurIPS 2025 — Zettelkasten-style notes with LLM-generated keywords/tags/links and *memory evolution* where new notes update old ones), Zep/Graphiti (temporal knowledge graph). The **Agent Skills** format is the clearest emerging *standard* (open, cross-vendor). There is not yet a settled standard for the memory *write* interface itself.

**Open research directions.**
- **Learned memory management (RL for memory decisions).** Memory-R1 (Yan et al., arXiv:2508.19828, ACL 2026) trains a Memory Manager (ADD/UPDATE/DELETE/NOOP) and an Answer Agent with PPO/GRPO, rewarding downstream QA accuracy rather than hand-labeled operations — outperforming Mem0/LangMem/A-Mem/Zep baselines with only 152 training QA pairs across LoCoMo, MSC, and LongMemEval. The strongest signal that *what to remember* is a learnable policy, not a fixed heuristic.
- **Context-as-playbook.** ACE (Zhang, Hu et al., Stanford/SambaNova/Berkeley; arXiv:2510.04618, Oct 2025) treats context as an evolving playbook maintained by a Generator/Reflector/Curator division of labor, using *delta updates* to avoid "brevity bias" and "context collapse" — matching a production GPT-4.1 agent (IBM CUGA) on AppWorld using the smaller DeepSeek-V3.1, reporting +10.6% on agent tasks with ~86.9% lower adaptation latency.
- **Parametric memory / continual learning.** The deepest question: whether durable knowledge belongs in *weights* (fine-tuning, continual learning) or *external memory*. The dominant 2026 answer is external — ACE explicitly positions context engineering as a first-class alternative to weight updates (cheaper, inspectable, reversible) — but the boundary is an active frontier.
- **Shared self-managed memory in multi-agent systems** raises both a capability (collective learning through shared blocks) and a risk (contagious poisoning), and is under-studied.

**The long-term vision.** The through-line from MemGPT to sleep-time compute to Letta Code is agents that genuinely *learn and evolve over time* — that spend idle cycles thinking, accumulate a durable skill library, and carry identity and continuity across sessions. As Letta's founders put it, today's models are "fundamentally stateless"; the goal of self-managed memory is to make statefulness — and therefore learning — a native property of agents.

## Synthesis of the memory arc
The four chapters of Part IV compose into a single architecture:
- **The Memory Taxonomy** gave the *kinds* of memory (working/episodic/semantic/procedural), the substrates, and the lifecycle — the vocabulary.
- **Prompt Caching and the Economics of Remembering** gave the *economics of keeping things in context* — the KV-cache reality that makes "what's hot vs cold" a cost decision. Self-management inherits this: every in-context memory block and re-read memory file has a token price, and context editing exists to reconcile memory with cache.
- **Retrieval as Memory** gave the *read side* — embeddings, hybrid search, reranking, agentic retrieval — the machinery for getting the right memory back into context at the right time.
- **Self-Managed Memory** (this chapter) gave the *write side in the agent's own hands* — deciding what to remember, editing and consolidating it, doing that work autonomously, increasingly offline.

The complete 2026 picture: an agent has **working memory** (its context window, governed by caching economics), **long-term memory** (external blocks, files, vector and graph stores), it **retrieves** relevant information into context on the read side, and it **manages its own memory over time** on the write side — writing, reflecting, consolidating, forgetting, often via a background sleep-time process. Not four separate systems but one loop: write quality bounds retrieval quality, retrieval surfaces what was written, caching prices what stays hot, and consolidation continuously reshapes the whole.

The key open questions are honest and unresolved: **How autonomous should memory be?** (Evidence favors scoped autonomy.) **Overwrite or accumulate?** (Letta's `replace` vs Mem0's ADD-only — unsettled.) **How do we know self-managed memory is helping?** (Write-side evaluation is immature; QA accuracy hides upstream errors.) **How do we secure it?** (MINJA shows the attack surface is real and defenses immature.) **Weights or external store?** (Currently external, but continual learning may shift the line.)

The trajectory is clear. Memory is the foundation for the thing the book is building toward: agents that don't just execute a loop but *persist and improve* across loops. A stateless model answers the same question the same way forever; an agent with good self-managed memory answers it better next time because it remembered what happened last time. That — memory as the substrate of learning — is why this is the capstone, and why getting the write side right is the next hard problem for the field.

## Recommendations (staged)
1. **Choose your paradigm by requirement, not hype.** Need *personalization with predictable cost* and bolting memory onto an existing stack → start with an externally-managed layer (Mem0, LangMem background manager); benchmark write latency and retrieval p95 first (LangMem's ~59.82s p95 vs Mem0's ~0.200s). Need *evolving, agent-curated state* across long-running sessions (assistants, coding agents, multi-week projects) → adopt self-managed memory (Letta blocks or Claude's memory tool), accepting the per-turn token/latency cost as the price of adaptivity. Move up the autonomy ladder when a fixed pipeline is producing stale/generic personalization and you're hand-tuning extraction rules per case.
2. **Constrain the write side from day one.** Enforce hard size caps per block/file (Anthropic's guidance: ≤~64KB/file, ≤~10MB/user) and per-user path/namespace isolation; validate all paths against traversal (`Path.resolve()` + `relative_to()`). Tag memories with provenance and (high-stakes) require a validation step before candidate memories are promoted to durable/trusted. Treat all query-derived memory as untrusted (MINJA). Filter sensitive data on write; document retention for compliance.
3. **Add higher-order operations, offline.** Introduce consolidation/reflection as a *background* job (Letta sleep-time agent, Claude Auto Dream, or a custom batch), not in the hot path, to avoid user-facing latency; trigger on idle time or session-end. Decide overwrite-vs-accumulate deliberately per field: overwrite volatile facts (current job, location) via replace; accumulate where history matters (preferences over time) with temporal tags.
4. **Evaluate the write side directly, then iterate.** Don't rely on end-to-end QA accuracy alone; it hides upstream errors. Run operation-level evaluation (HaluMem-style extraction recall/accuracy and update omission/hallucination) plus LoCoMo/LongMemEval for read-side regression. Benchmarks that change the approach: high update-omission → consolidation too conservative; low extraction accuracy → tighten what the agent may write; a successful MINJA-style injection → add isolation/provenance before shipping persistent memory to multiple users. Consider learned management (Memory-R1-style RL, or ACE-style playbook curation) once you have a reward signal (downstream task success) and enough traffic to train on.

## Caveats
- The field moves fast and many key numbers are vendor-reported: Anthropic's 39%/84% context-management figures and Mem0's LoCoMo/LongMemEval scores are self-reported internal or first-party evaluations; treat as directional, not independent. Version churn is high (Letta v1 agent, Mem0's move to ADD-only, Claude Code auto-memory Feb 2026) — verify against current docs.
- Best practices are genuinely contested: overwrite (`core_memory_replace`) vs accumulate (Mem0 ADD-only), hot-path vs background writing, full vs scoped autonomy are live debates. This chapter takes the position that scoped autonomy + background consolidation + direct write-side evaluation is the current pragmatic optimum, but reasonable architects disagree.
- Security and privacy research is early: MINJA and follow-ons demonstrate attacks convincingly; robust deployed defenses are less mature. Persistent self-managed memory in multi-user or multi-agent settings should be treated as a security-sensitive subsystem.
- Write-quality evaluation is immature: the write-side benchmarks cited (HaluMem, RealMem, and the 2026 error-attribution cluster such as MemTrace, MemSyco-Bench, MemoryRewardBench, Fine-Mem) are new and not yet consensus standards; several were surfaced via citation lists — verify specific numbers against the primary arXiv sources before quoting as established.
