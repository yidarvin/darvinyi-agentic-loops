# The Memory Taxonomy

Research reference for *Agentic Loops*, Chapter 15 (opens Part IV, Memory; followed by "Prompt Caching and the Economics of Remembering," "Retrieval as Memory," and "Self-Managed Memory"). Current as of 2026. The reader already understands the agent loop, context economics, tool calls, MCP, skills, and multi-agent systems. This chapter establishes the full conceptual map and taxonomy of agent memory deeply, but leaves caching, retrieval/RAG mechanics, and self-editing memory to the later chapters. Version-gated framework details drift; version-pin and re-verify. Note that many memory-benchmark figures are vendor-reported.

## TL;DR
- LLMs are stateless: every API call is independent, so an agent's only "live" memory is its context window. **Agent memory is the engineering discipline of persisting information outside the context window and retrieving the right pieces back into it** — the solution to both statelessness and the finite context window.
- The field converged on a cognitive-science taxonomy formalized by CoALA (Sumers, Yao, Narasimhan & Griffiths, TMLR 02/2024): one **short-term working memory** (the context window) plus three **long-term** stores — **episodic** (experiences/events), **semantic** (facts/knowledge), and **procedural** (skills/how-to, held in prompts, code, and weights). This is the dominant lens across LangMem, Letta, Mem0, and Zep.
- These types map onto concrete substrates (context window, vector stores, knowledge graphs, files) and a lifecycle of operations (write/encode, store, retrieve, forget/update). The real open questions are not "how do we store more" but "how do we store less, better" — memory can pollute context, go stale, and leak private data, and much of what looks like a "memory problem" is really a context-management or retrieval problem.

## Why memory matters: statelessness and the finite context window
The starting point: **the LLM at the center of an agent is a stateless function.** Each API call is independent; the model retains nothing from the previous call. The illusion of a continuous conversation is produced entirely by the harness re-sending the accumulated message history every turn. There is no hidden server-side state in the model; the "conversation" lives in the client's messages array.

Two consequences. First, the **only** thing the model can attend to is what's currently in its context window — if information isn't in the window, it doesn't exist for that call. Second, the window is finite and its *effective* capacity is smaller than nominal: attention isn't uniform, and performance degrades as context grows longer and noisier ("lost-in-the-middle," "context rot"). Anthropic's framing: "context windows of all sizes will be subject to context pollution and information relevance concerns," so the solution isn't simply larger windows.

Two things both get called "memory": **parametric knowledge** (learned during training, baked into weights, frozen, vast but static, hard to inspect/update/attribute) and **agent memory** (dynamic, updatable, external state the agent writes and reads over its lifetime — what *this* user said, what *this* task required, what worked last time). A production definition this chapter adopts: **agent memory is persistent external state that can be written, updated, deleted, retrieved, and injected into the model context to improve future behavior under cost, latency, and safety constraints.** The context window is *working* memory; it exists only during the current inference request. Memory answers two coupled problems: **statelessness** (persist across turns/sessions/tasks) and **finite/degrading context** (keep the window small and relevant, holding the rest in retrievable stores).

## Short-term vs long-term
**Short-term / working memory = the context window.** Holds the current conversation, recent turns, tool results, active task state. Immediately accessible, zero retrieval latency, but finite and ephemeral — resent every call, costs tokens every call, lost at session end or truncation/compaction.

**Long-term memory = persistent storage outside the window** — databases, vector stores, knowledge graphs, files. Effectively unlimited, durable across sessions, but inert until *retrieved*. Nothing in long-term memory affects a generation unless first pulled into the window, and retrieval has latency, cost, and imperfect precision/recall.

The interplay is the whole game. CoALA describes working memory as "the central hub connecting different components of a language agent" — where perceptual inputs, retrieved knowledge, and reasoning outputs meet. CoALA's three internal action types are exactly the movements between tiers: **retrieval** ("read from long-term memory") reads into working memory, **reasoning** "reads from and writes to working memory," and **learning** "write[s] to long-term memory."

## The cognitive-science taxonomy (the heart of the chapter)
The formalizing paper is **CoALA — "Cognitive Architectures for Language Agents"** (Sumers, Yao, Narasimhan, Griffiths, Princeton; TMLR 02/2024; arXiv:2309.02427): agents "organize information ... into multiple memory modules ... short-term working memory and several long-term memories: episodic, semantic, and procedural." Borrowed from Tulving's episodic/semantic distinction plus procedural and working memory.

**Episodic — experiences and events.** CoALA: *"Episodic memory stores experience from earlier decision cycles."* The log of what happened — past conversations, task trajectories, specific interactions, input-output pairs. Answers "what happened / what did we do?" Canonical uses: recalling what a user said before, and supplying **few-shot examples from past successful episodes** (LangMem: episodic memory "often takes the form of few-shot examples derived from past successful interactions"). Instance-specific, preserves original context (when/where/why).

**Semantic — facts and knowledge.** CoALA: *"Semantic memory stores an agent's knowledge about the world and itself."* Generalized, decontextualized: the user's name, preferences, domain facts, a knowledge base, a user profile. Answers "what is true?" LangMem stores facts/user info/knowledge triples; Mem0 extracts "User prefers vegetarian restaurants"; MemMachine calls this "Profile Memory." The key conceptual move: the *event* "user told me yesterday she's vegetarian" (episodic) vs the *fact* "user is vegetarian" (semantic). The transformation between them is **consolidation**.

**Procedural — skills and how-to.** Most misunderstood, because in agents it's largely *not* retrieved data. CoALA: *"Language agents contain two forms of procedural memory: implicit knowledge stored in the LLM weights, and explicit knowledge written in the agent's code."* It's *how the agent does things* — system prompt, tool-use patterns, learned workflows, decision code. LangMem: procedural memory "is saved across a combination of model weights, agent code, and agent's prompt that collectively determine the agent's function." CoALA warns writing to procedural memory (editing code or fine-tuning) is "significantly riskier than writing to episodic or semantic memory," and it "must be initialized by the designer" whereas episodic/semantic may start empty. Connects to earlier chapters: **skills are procedural memory made explicit and portable.**

**Working memory** is the fourth type (short-term, above): CoALA's store that "maintains active and readily available information as symbolic variables for the current decision cycle."

Why this framing won: each type maps to a *different engineering problem* with a different substrate, update cadence, and failure mode — episodic, semantic, and procedural memories "operate differently, decay differently, and fail differently." The taxonomy is descriptive, not prescriptive: it names gaps so they become visible. Limits (see below): CoALA classifies episodic and semantic both as "long-term memory" without formally distinguishing their persistence/decay semantics.

Intellectual lineage also runs through **Generative Agents** (Park et al., Stanford/Google, 2023), which introduced the **memory stream** — an append-only, timestamped, natural-language log — with a retrieval function scoring by a weighted sum of **recency** (exponential decay), **importance** (LLM-assigned 1–10 at creation), and **relevance** (embedding cosine similarity), plus **reflection** (periodic synthesis of raw observations into higher-level insights, triggered when summed importance of recent events exceeds a threshold — 150 in the original, ~2–3 reflections per "day"). This trio — memory stream, scored retrieval, reflection — is the reference design most later systems recapitulate.

## Mechanisms and substrates
- **Context window (working memory).** The messages array. "Just keep it in the prompt." Fast, no retrieval step, token-costly and lossy at scale. Managed by truncation/summarization/context editing rather than a store.
- **Vector stores / embeddings (episodic + semantic long-term).** Memories embedded, retrieved by similarity (top-K cosine). Dominant RAG-style mechanism; backbone of Mem0, LangMem, most production systems. Backends: Qdrant, Pinecone, Chroma, Weaviate, pgvector, Milvus. Strengths: scales, fuzzy/semantic recall, cheap. Weaknesses: no native time or contradiction handling, retrieval quality degrades as the store grows, pure similarity can surface stale/irrelevant items.
- **Structured stores / knowledge graphs (semantic).** Facts as entities and relationships. **Zep's Graphiti** is the leading example: a **bi-temporal knowledge graph** recording both when an event occurred and when it was ingested; three tiers (episode subgraph = raw messages; semantic-entity subgraph = extracted facts with validity windows; community subgraph = cluster summaries); when a new fact contradicts an old one, it *invalidates* the old by closing its validity window rather than deleting. Enables point-in-time queries and contradiction-free current-state answers. Retrieval fuses semantic embeddings, BM25, and graph traversal, avoiding LLM calls at query time; Zep Cloud advertises <200ms retrieval. Strengths: temporal reasoning, auditable provenance, multi-hop. Weaknesses: heavier write-time processing, higher storage cost; immediate post-ingestion retrieval can lag until background graph processing completes.
- **Files / documents (various).** Memory as files the agent reads/writes. (1) **MemGPT/Letta's** OS-inspired paging between in-context and external storage; (2) **Claude's memory tool** (`memory_20250818`, launched Sept 29 2025), a client-side file directory (`/memories`) with `view/create/str_replace/insert/delete/rename`, where the model issues instructions and the developer's infra executes them; (3) **Manus's** "filesystem as the ultimate context" (unlimited, persistent, restorable; drop a page's content but keep the URL; the todo.md scratchpad for attention management). Strengths: unbounded, human-inspectable, agent-controllable, no embedding infra. Weaknesses: retrieval by path/grep not semantics; organization depends on agent discipline.
- **Summaries / compression (working-to-long-term consolidation).** Summarizing a conversation to persist its gist — the **consolidation** process (verbose episodic → compact semantic/notes). ChatGPT's "dreaming" and Letta's sleep-time reflection are consolidation engines. Strength: bounds growth. Weakness: lossy.
- **Key-value / document stores (semantic, user profiles).** LangGraph's BaseStore persists long-term memories as JSON by namespace and key. Simple, fast, exact-match; ideal for structured profiles/preferences.

Substrates aren't mutually exclusive: Mem0 combines a vector store with an optional graph; Zep fuses vectors + keyword + graph; production stacks run RAG over a docs corpus *and* a memory layer over interactions, because they solve different problems.

## Memory operations — the lifecycle
Every memory system implements some subset of a common read-write lifecycle:
- **Writing / encoding.** *Explicit* (user/agent says "remember X," or a tool call) or *automatic* (background extraction of salient facts). Mem0 extracts candidate facts via an LLM call; LangMem offers hot-path tools the agent calls plus background managers extracting asynchronously. Central design question: what *counts* as memory — over-broad extraction produces noisy, polluted stores.
- **Storage.** The substrate choice, plus metadata (timestamps, user/session/agent IDs, importance scores) and scoping/namespacing.
- **Retrieval.** By similarity, recency, importance, relevance, or a weighted combination (Generative Agents' recency+importance+relevance; Zep's hybrid fusion). *Query-based* (search on the current turn) or *proactive* (always-inject core memories). LangMem distinguishes data-independent memories (always in the prompt, e.g., core procedural rules) from data-dependent ones (recalled by similarity/context).
- **Forgetting / updating / consolidation.** Removing outdated info, reconciling changed facts, merging duplicates, bounding growth — where systems most differ. Mem0's classic pipeline retrieves similar memories and has an LLM choose **ADD / UPDATE / DELETE / NOOP** per candidate fact; its 2026 redesign moved to a single-pass **ADD-only** model that never overwrites, preserving the full historical sequence and letting the app interpret the latest value by `created_at`. Zep never deletes — it invalidates. LangMem consolidates by merging related memories and resolving contradictions to avoid "memory hoarding." The overwrite (clean current state) vs append (full auditable history) fork is a genuine design decision.

## Architectural patterns
- **The memory-hierarchy analogy (CPU cache / RAM / disk).** Introduced by **MemGPT** (Packer et al., UC Berkeley, 2023, "Towards LLMs as Operating Systems"). Context window = RAM, external storage = disk; the LLM gets OS-style **virtual context management**, *paging* between a **main context** (system instructions, FIFO message queue, writable scratchpad) and **external context** (recall storage = searchable history; archival storage = vector-indexed long-term store), via self-issued function calls (`archival_memory_search`, `core_memory_append`, etc.). An interrupt/"heartbeat" hands control to the agent for multiple memory operations before responding. The model manages its own hierarchy, providing "the illusion of larger context resources."
- **Hot vs cold / tiered memory.** Frequently-accessed items stay "hot" (in/near context); rarely-used items live "cold" externally, paged in on demand. Claude's context editing (`clear_tool_uses`) + memory tool is a hot/cold system: stale tool results cleared while insights preserved to files.
- **In-context vs retrieval-based vs parametric.** Three loci: window (fast, ephemeral, token-costly), external store (retrievable, inspectable, updatable), or weights (parametric — seamless but "hard to audit ... hard to delete from ... and expensive to update," why most deployed agents favor non-parametric stores).
- **Memory scoping.** Namespaced — per-user (most common, prevents cross-user leakage), per-session, per-agent, or **shared across agents**. LangMem namespaces every memory (typically by user_id); memories can scope to app routes, users, teams, or learned core procedures across all users.
- **Single-agent vs org-level memory in multi-agent systems.** Isolated (each subagent its own context to prevent cross-contamination — Manus's rationale) or shared (a common store multiple agents read/write). **Letta's shared memory blocks** let multiple agents attach to the same block and see the same value. Letta's **sleep-time agents** are a special case: a background agent shares memory blocks with a primary agent and reorganizes them asynchronously (the primary handles conversation and searches external memory; the sleep-time agent alone holds tools to edit the primary's in-context blocks). Shared state is both a coordination mechanism and a failure vector (one agent's context pollution can cascade).

## The current landscape (2026)
- **MemGPT / Letta.** OS-memory-hierarchy pioneer, rebranded Letta as a general stateful-agent framework. Main/recall/archival tiers; **self-editing memory** via tool calls; **memory blocks** (labeled, character-limited context sections the model edits — original "Human"/"Persona" blocks); **sleep-time compute** (background consolidation during idle turns); shared blocks for multi-agent memory. (Self-editing memory gets its own chapter.)
- **Mem0.** Most-adopted drop-in memory layer. Lightweight `add / search / update / delete` over an embeddings store with LLM-extracted facts, scoped per user/session/agent, optional graph mode (Mem0g, Neo4j). Complementary to RAG, not a replacement. ECAI 2025 paper (Chhikara et al., arXiv:2504.19413): 26% relative LLM-as-judge accuracy gain over OpenAI's memory on LoCoMo, ~91% lower p95 latency, ~90% fewer tokens (~1.8K vs ~26K per conversation). 2026 token-efficient algorithm: 91.6 on LoCoMo, 93.4 on LongMemEval at under ~7K tokens/retrieval.
- **Zep / Graphiti.** Temporal-knowledge-graph approach. Bi-temporal facts with validity windows, automatic invalidation, episode-level provenance, hybrid retrieval, <200ms. Zep paper (Rasmussen et al., arXiv:2501.13956) beats MemGPT on Deep Memory Retrieval (**94.8% vs 93.4%**), reports LongMemEval accuracy up to **18.5%** improvement with **90% lower latency**, ~1.6K tokens/response vs 115K for full context, with especially large gains on temporal reasoning (+38.4%) and multi-session (+30.7%). Open-source core (Graphiti) + enterprise platform (Zep).
- **LangMem / LangGraph memory.** Most explicitly taxonomy-driven. Semantic/episodic/procedural as first-class types; hot-path tools vs background managers; backend-agnostic store abstraction (BaseStore for cross-thread memory, checkpointer for in-thread session state); consolidation to prevent unbounded growth.
- **Claude's memory tool + context editing (Anthropic).** File-based, client-side, developer-controlled. CRUD commands paired with context editing that clears stale tool results. Per the Sept 29 2025 announcement, combining the memory tool with context editing **improved performance 39% over baseline** (context editing alone: **29%**), and in a 100-turn web-search eval context editing **reduced token consumption 84%**.
- **ChatGPT memory (OpenAI).** Two mechanisms: **saved memories** (explicit facts, injected into every chat via a "Model Set Context" section) and **reference chat history** (a synthesized profile from past conversations, launched April 10 2025). A "dreaming" background process curates/updates memories over time.

Standardization is emerging but incomplete: the four-type taxonomy is a shared vocabulary and `add/search/update/delete` a de-facto operation set, but there's no standard storage format, retrieval protocol, or interoperability layer. Memory is increasingly managed infrastructure rather than a prompt-engineering afterthought.

## Open questions and debates
- **How much memory is useful vs harmful.** More is not monotonically better. Retrieved memories consume tokens, can surface irrelevant/stale info, and can degrade performance (context pollution/rot). Real tension between remembering everything and staying focused; the window should be protected, not filled. Anthropic's position: the fix is better *management*, not bigger windows.
- **When memory is over-engineering.** A large fraction of apparent "memory needs" are met by good context management or plain retrieval over a source-of-truth corpus. LangMem's guidance: if knowledge lives in a docs store that's the source of truth, retrieve over it directly rather than building a semantic-memory system. "Start simple" applies.
- **Learned/parametric vs external/retrieved.** Weights (fine-tuning, adapters, knowledge editing like ROME/MEMIT) vs external stores. Parametric is seamless but opaque, hard to delete from (unlearning is immature), costly to update; external is inspectable/updatable/attributable. Deployed agents currently favor external stores; parametric approaches remain active research.
- **Reliability of extraction and retrieval.** Automatic extraction is imperfect (misses salient facts, stores noise, mis-resolves contradictions); retrieval can fail to surface the right memory even when it exists. These compound in long-horizon and multi-agent settings.
- **Privacy and security.** Persisting user data long-term triggers GDPR/data-protection obligations and creates an attack surface. **Memory poisoning** (MINJA, NeurIPS 2025) shows adversaries can inject malicious records via query-only interaction; poisoned memories persist across sessions and can cascade in multi-agent systems. Per-user namespacing, provenance tracking, and write-filtering of sensitive fields are baseline mitigations.
- **Evaluation.** **LongMemEval** (Wu et al., arXiv:2410.10813, ICLR 2025; 500 questions over scalable chat histories testing information extraction, multi-session reasoning, temporal reasoning, knowledge updates, abstention) and **LoCoMo** (Maharana et al., ACL 2024; ~10 very long multi-session dialogues, ~600 turns and ~16–26K tokens each). Sobering headline: per LongMemEval, "commercial chat assistants and long-context LLMs show[] a 30% accuracy drop on memorizing information across sustained interactions," with long-context LLMs showing a **30%–60% performance decline** on the ~115K-token LongMemEval_S setting vs oracle retrieval, and GPT-4o reaching only 30%–70% accuracy in a simpler setting. Even the strongest 2026 systems saturate easy categories (single-session, knowledge-update) while multi-session and temporal reasoning remain materially harder. Read benchmark numbers skeptically: vendors report on their own platforms with proprietary optimizations, judge inconsistency introduces error bars, and some scores depend on incidental factors (e.g., default embedding models). Full-context baselines consistently consuming ~23–26K tokens/query illustrate why token cost must be reported alongside accuracy — good retrieval achieves comparable accuracy at >90% token reduction (~600–1,500 tokens/query).

## Recommendations (staged)
1. **Start with context management, not a memory system.** Before adding external memory, exhaust cheaper options: trim/structure the prompt, summarize/compact long histories, use context editing to clear stale tool results. If single-session or the knowledge lives in an existing source-of-truth corpus, retrieve over that corpus directly. Escalate when you see cross-session needs (users repeating themselves), context-window exhaustion on long tasks, or personalization requirements.
2. **Add long-term memory scoped to the actual need, typed by the taxonomy.** Facts about users → **semantic memory** (vector/KV store; Mem0, LangMem, BaseStore; namespace per user). Learn from past interactions (few-shot from successful episodes) → **episodic memory** (timestamped records, retrieved by relevance+recency). Consistent behavior/workflows → **procedural memory** in the system prompt, skills, and code — not a vector store; consider fine-tuning only when prompt/skill approaches plateau (heed CoALA's warning that procedural writes are riskiest).
3. **Choose the substrate by the dominant query pattern.** Fuzzy semantic recall, simplest integration → **vector store** (Mem0-style). Facts that **change over time**, contradiction handling, point-in-time/temporal queries, auditability → **temporal knowledge graph** (Zep/Graphiti); escalate when temporal/knowledge-update accuracy is unacceptable with a flat vector store (independent 2026 comparison: Zep 63.8% vs Mem0 49.0% on LongMemEval). Agent-controlled, inspectable, unbounded scratchpad and long-horizon task state → **files** (Claude memory tool / MemGPT-Letta / Manus). Self-editing, stateful, long-lived personalized agent → **Letta memory blocks** with sleep-time consolidation.
4. **Instrument and evaluate.** Adopt a benchmark (LongMemEval, LoCoMo) or a domain analog, tracking **accuracy, token cost per query, and latency** together. A system with high accuracy but ~26K tokens/query isn't production-viable; low latency but poor recall is useless. Revisit architecture when multi-session/temporal accuracy stalls, token cost per query climbs, or you see stale/contradictory answers.

**Cross-cutting guardrails:** namespace per user; track provenance so a bad memory can be traced and removed; filter sensitive fields before writing; prefer append-with-timestamp over destructive overwrite when auditability matters; give the agent explicit instructions about what to write (under-specified extraction produces polluted stores).

## Caveats
- Vendor benchmarks are self-reported: Mem0's and Zep's headline numbers come from their own papers/platforms, often against a self-defined "full-context" baseline, with LLM-as-judge scoring (~±1 point noise) and proprietary managed-platform optimizations not in the open-source builds. Treat cross-vendor comparisons as directional; independent audits have shown some scores hinge on incidental factors (one local-first system's high LongMemEval Recall@5 was attributed largely to its default embedding model rather than its advertised architecture).
- The taxonomy is a lens, not a law: episodic/semantic/procedural is a useful organizing device, but boundaries blur (an episodic record consolidates into a semantic fact; procedural memory spans prompt, code, and weights), and CoALA doesn't formally distinguish the decay/persistence semantics of episodic vs semantic.
- This chapter is a map, not a manual: retrieval/RAG mechanics, prompt caching economics, and self-managed/self-editing memory each get dedicated treatment in the following chapters; framework internals here are deliberately summarized.
- The landscape moves fast: framework specifics (Mem0's extraction algorithm, Claude's tool versions, ChatGPT's memory architecture, Letta's sleep-time features) changed repeatedly through 2025–2026 and will continue to; treat version-specific details as snapshots.
