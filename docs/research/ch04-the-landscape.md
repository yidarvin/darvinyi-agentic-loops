# The Landscape: Claude Code, OpenAI Codex, and opencode as Points in a Design Space

Research reference for *Agentic Loops*, Chapter 4. Current as of 2026. This is a design-space analysis, not a feature list. Version numbers, model names, star/MAU counts, and pricing are 2026 snapshots and drift fast; version-pin at build time.

## TL;DR
- Agentic coding tools are converging on one architecture — a simple model-in-a-loop wrapped in a large deterministic harness — and diverging almost entirely on packaging. The MBZUAI reverse-engineering of Claude Code v2.1.88 (arXiv:2604.14228) quantifies the split: roughly 1.6% of the codebase is AI decision logic, ~98.4% operational harness. The interesting design decisions are all in that 98.4%.
- The three reference tools stake out three corners: Claude Code is local, interactive, collaborative, single-vendor (Anthropic); OpenAI Codex's defining bet is cloud/async delegation with OS-level sandboxing; opencode is open-source, terminal-first, model-agnostic (75+ providers) with a persistent client/server daemon. They optimize, respectively, for trusted human-in-the-loop control, parallel fire-and-forget throughput, and vendor independence.
- For expert readers the actionable lesson: pick your position on ~7 axes (execution locus, model coupling, autonomy, context strategy, extensibility, permission/safety model, interface), because as frontier models converge in raw capability the harness — not the model — is the differentiator and the moat.

## The design-space framing (the real content of the chapter)
An agentic coding tool minimally (a) runs an agent loop, (b) exposes a tool interface (file/shell/search/web), (c) manages context, and (d) enforces control (permissions, sandboxing). Everything past that minimum is a position on one axis:

**Axis 1 — Execution locus (local/interactive ↔ cloud/async/delegated).** Does the loop iterate on your machine against your live tree, or in a remote sandbox you fire work at? Largest downstream blast radius: cascades into latency, parallelism, trust, network posture, interaction model.

**Axis 2 — Model coupling (single-vendor/closed ↔ model-agnostic/open).** Bonded to one lab's models or treats the model as swappable? Coupling buys co-design and first-party caching at the cost of lock-in; agnosticism buys insurance and per-task routing at the cost of model-specific tuning.

**Axis 3 — Autonomy level (collaborative pair ↔ autonomous delegate).** "Asks before every action" ↔ "assign a ticket and walk away." Partly a permission setting, partly a product philosophy shaping onboarding, interface, metrics.

**Axis 4 — Context strategy.** Persistent memory files, compaction/summarization, subagent isolation, repo-mapping/retrieval, lazy loading of instructions and tool schemas.

**Axis 5 — Extensibility model.** How capability is added and at what context cost: lifecycle hooks (zero cost) → skills (on-demand) → plugins (packaged) → MCP servers (schema cost) → provider plugins (model-agnostic).

**Axis 6 — Permission/safety model.** Where the trust boundary sits and whether it fails open or closed: permission rules (fail-closed deny-first), OS-level sandboxing (kernel), container isolation, git rollback. Best systems layer several.

**Axis 7 — Interface (terminal TUI ↔ IDE ↔ web/desktop).** Terminal wins on SSH/remote/CI and proximity to shell/git; IDE wins on diff review and greenfield visual work; web/desktop suits supervising many parallel async agents.

Axes are not independent (execution locus drags autonomy and safety with it) but they are the right coordinate system for placing any tool.

## Claude Code (the spine)
**"Small loop, large harness."** The April 2026 MBZUAI/VILA-Lab paper *Dive into Claude Code* (arXiv:2604.14228, analyzing v2.1.88, ~1,900 TS files, ~512K lines) cites a community estimate that "only about 1.6% of Claude Code's codebase constitutes AI decision logic, with the remaining 98.4% being operational infrastructure." (Caveat: this is a cited community estimate on a leak-derived, partly-generated bundle, not the authors' primary line-count — directional topline, corroborated qualitatively by Anthropic and Boris Cherny.) The core is "a simple while-loop that calls the model, runs tools, and repeats" — a single `queryLoop()` async generator in `query.ts`, single-threaded with a flat message history. Principle 7, "minimal scaffolding, maximal operational harness": invest in deterministic infrastructure rather than decision scaffolding, on the premise that capable models "benefit more from a rich operational environment than from frameworks that constrain their choices." Security corollary: "because reasoning and enforcement occupy separate code paths, a compromised or adversarially manipulated model cannot override the sandboxing, permission checks, or deny-first rules." The paper grounds design in five human values and thirteen principles; seven functional components (User, Interfaces, Agent loop, Permission system, Tools up to 54 built-in, State & persistence via append-only JSONL, Execution environment).

**Posture:** local/interactive/collaborative. Runs in your terminal against your files, shell, git. Works across files. Each loop iteration calls the Anthropic API (network required); all tools run locally.

**Permission model (7 modes):** `plan`, `default`, `acceptEdits`, `auto`, `dontAsk`, `bypassPermissions`, plus internal `bubble`. Three core modes cycle with Shift+Tab. `auto` (research preview, Sonnet 4.6/Opus 4.6+ on the API, Team/Enterprise gating) removes prompts but routes each shell/network action through an ML safety classifier; Anthropic's rationale: "telemetry showed users approved roughly 93% of permission prompts," and the classifier showed low false-positive but higher false-negative rates on dangerous actions (so it is not a substitute for review). In headless `-p`, 3 consecutive or 20 total classifier blocks abort. `dontAsk` denies everything not allow-listed. `bypassPermissions` skips prompts but deny rules, ask rules, and hooks still fire. Enforced fail-closed: deny rules win in every mode. Ships an open-source sandbox-runtime (Seatbelt on macOS, bubblewrap on Linux, HTTP/SOCKS proxy).

**Context management:** CLAUDE.md (hierarchical additive memory, lazy-loaded nested files); auto memory (MEMORY.md, first 200 lines / 25KB at start); **five-stage compaction pipeline** (budget reduction → snip → microcompact → context collapse → auto-compact), each running only when cheaper stages are insufficient, circuit-breaker after 3 consecutive failures; subagents run isolated loops returning only summaries (~7x the tokens of a standard turn).

**Extensibility (five mechanisms by context cost):** CLAUDE.md (always-on) < hooks (zero cost, event-triggered, the only unconditional block via PreToolUse exit code 2) < skills (on-demand markdown, unified with slash commands in 2026) < plugins (packaged, namespaced, versioned) < MCP (external services, highest cost). Subagents form a delegation layer; Feb 2026 "agent teams" (with Opus 4.6) adds coordinated sessions with peer messaging. Hooks span 27 event types.

**Headless:** `claude -p` runs the full loop non-interactively with `--output-format text|json|stream-json`, stdin piping (10MB cap), `--allowedTools`, `--permission-mode`, `--max-turns`, `--bare` (skip auto-discovery for reproducible CI). Exposed as the Claude Agent SDK (Python/TypeScript).

**Optimizes for** trusted, inspectable, human-in-the-loop local collaboration. Bets: model is a commodity, invest in the harness; single-threaded flat history beats multi-agent cleverness (inspectable, auditable); fail-closed everywhere; cache economics first-class; single-vendor Anthropic coupling (enforced in 2026).

## OpenAI Codex (the 2025–2026 agent, not the 2021 model)
Umbrella over several surfaces sharing one account, model family, and engine: cloud web agent (chatgpt.com/codex, preview May 16 2025, GA Oct 6 2025), open-source CLI (first April 16 2025), IDE extensions (VS Code, Cursor, Windsurf), macOS/Windows desktop app. Growth: OpenAI cited ">2M weekly users" March 2026 (>70% MoM), ">4M weekly developers by April 21 2026," a June 2 2026 report put WAU past 5M with users "increasingly running multiple Codex tasks in parallel."

**CLI architecture:** rewritten from TypeScript to **Rust** (repo now ~95% Rust; TS survives as a thin npm launcher). Reasons (co-lead Fouad Matin, Discussion #1174): zero-dependency install, native security bindings, no GC pauses, a wire protocol for other-language extension. Event-driven: a queryLoop-equivalent core emits lifecycle events (TurnStarted, ExecCommandBegin/End, PatchApplied, TokensUsed) consumed by TUI/IDE as pure subscribers. Tool dispatch via ToolRouter/ToolOrchestrator (approval → sandbox selection → execute → retry-with-escalation). Distinctive tools: **`apply_patch`** (structured edits via a formal Lark grammar, validated before application), a persistent Node.js `js_repl`, BM25 `tool_search`, `spawn_agent`/`wait_agent` hierarchical subagents. Sessions are **persistent SQLite-backed threads** (resumable, forkable, archivable, rollback-able) — explicit contrast with Claude Code's per-session-fresh model. Responses API only; default a GPT-5.x-Codex variant (272K–400K context).

**OS-level sandboxing (the safety spine):** every model-issued shell command runs in a kernel-enforced sandbox: **Seatbelt** (`sandbox-exec`, deny-by-default SBPL) on macOS; **bubblewrap + Landlock LSM + seccomp-BPF** on Linux (read-only by default, writable roots explicit, namespaces unshared); **restricted tokens + synthetic SIDs + dedicated sandbox users + firewall rules** on Windows (promoted from experimental in v0.100.0, deep-dive May 13 2026). **Network disabled by default** inside the sandbox. Three sandbox modes (`read-only`, `workspace-write`, `danger-full-access`) plus an approval-policy layer. Cloud agent runs each task in an isolated container preloaded with the repo, historically with internet disabled.

**AGENTS.md:** a cross-tool markdown "README for agents" OpenAI introduced August 2025. Per the Linux Foundation (Dec 9 2025), "adopted by more than 60,000 open source projects and agent frameworks including Amp, Codex, Cursor, Devin, Factory, Gemini CLI, GitHub Copilot, Jules and VS Code," donated alongside Anthropic's MCP and Block's goose as a founding project of the Agentic AI Foundation.

**Reasoning effort:** `model_reasoning_effort` = minimal/low/medium/high/xhigh (via `-e`, `/effort`, profiles). Non-monotonic cost-quality: on GPT-5.5-Codex, semantic equivalence climbed 15%→42%→69%→88% low→xhigh, with **high** usually the best cost-quality point; other benchmarks found medium optimal for GPT-5.1-Codex-Max (high roughly doubled tokens, forcing compaction). Profiles (`fast`/`careful`/`auto`) bundle model + effort + tier.

**Philosophy — "a tool for editing files" and delegation.** Codex team (Hanson Wang, Alexander Embiricos): shift "from pairing to delegation," interacting like "asynchronous collaboration with a colleague" you assign tickets. OpenAI: "the asynchronous, multi-agent workflow introduced by Codex will become the de facto way engineers produce high-quality code." Successful users adopt an "abundance mindset," running many parallel tasks. Philosophically opposite to Claude Code's real-time steering.

**Optimizes for** throughput and delegation at scale, with security via containment rather than prompting. Bets: the future is many autonomous agents in parallel isolated environments; sandbox at the OS/kernel level so the agent moves fast inside a safe box; meet developers on every surface sharing one engine; OpenAI-first coupling with an open-source CLI and MCP support.

## opencode
**Origin/scale:** open-source, MIT, from the SST team (Anomaly Innovations — Dax Raad, Jay V, Frank Wang), first released June 19 2025. A June 8 2026 BetaKit interview disclosed 8M monthly active users and 172,000 GitHub stars ("ahead of Claude Code, ahead of Codex CLI"), with a projected $25M expected annual revenue. Repo: github.com/sst/opencode.

**Architecture — persistent client/server daemon** (its most distinctive feature vs Claude Code). Running `opencode` starts two processes: a background **server** (daemon) owning session state, history, and filesystem ops, and a **TUI client** connecting to it. The server persists between terminal sessions — close the terminal, SSH back, reconnect to the same live session with full context. (Claude Code and Aider lose session state on terminal close.) The server exposes an HTTP API (Hono) with a shared event bus over SSE, so the Bubble Tea TUI, a Tauri desktop app, and IDE extensions are all clients of one engine. Git snapshots at step-start for undo; sessions in SQLite. (Engine is primarily TypeScript on Bun; the TUI is Go/Bubble Tea — earlier "written in Go" coverage refers to the TUI client specifically.)

**Model-agnosticism — the defining bet.** Connects to **75+ providers** (Models.dev + Vercel AI SDK, any OpenAI-compatible endpoint, local via Ollama). BYOK, pay providers directly, no subscription. Switch models mid-session (cheap for exploration, frontier for hard refactors). The prompt pipeline selects provider-specific system prompts by matching model ID (Claude→anthropic.txt, GPT→beast.txt, GPT-5→codex_header.txt, Gemini→gemini.txt) and swaps tools by model (GPT gets `apply_patch`). The strategic payoff became concrete during the **Anthropic third-party-harness crackdown**: on Jan 9 2026 Anthropic blocked subscription OAuth tokens outside official Claude Code ("This credential is only authorized for use with Claude Code"), formalized the ToS prohibition in February, extended it to all third-party harnesses by April 4 2026. opencode shipped OpenAI OAuth via `/connect`, OpenAI officially partnered to extend Codex/ChatGPT subscription support inside opencode, and its stars roughly doubled. Model-agnosticism was insurance against platform-policy risk.

**AGENTS.md, Build vs Plan modes.** Reads AGENTS.md (same convention as Codex, so both coexist). Two primary agents toggled with Tab: **Plan** (read-only, edit/bash denied or ask) and **Build** (full access). Three built-in subagents (General, Explore, Scout) plus hidden system agents (compaction, title, summary). Configured in `opencode.json` or `.opencode/agent/*.md` with frontmatter, fine-grained per-tool **permissions** (`read`, `edit`, `bash`, `glob`, `grep`, `task`, `webfetch`; `.env` reads denied by default; `doom_loop` detection on the same call repeating 3x; `external_directory` gating). LSP integration feeds diagnostics back as grounding.

**Agent Client Protocol (ACP):** implements ACP server-side (`opencode acp`), an open JSON-RPC-over-stdio protocol standardizing editor↔agent communication, so Zed, Neovim (CodeCompanion, Avante), and JetBrains drive the same engine.

**Optimizes for** vendor independence, data sovereignty (no code on opencode servers with BYOK/local), and terminal/remote workflows. Bets: the harness is durable infrastructure, the model a swappable commodity — never bond to one vendor; a persistent server is the right primitive for long-running/multi-surface work; open-source + weekly cadence out-iterates closed tools; meet developers over SSH/tmux and any editor via ACP.

## Philosophical contrasts, made concrete
One task — *"find and fix the failing test in auth.test.ts, then open a PR"*:
- **Claude Code (local/interactive/collaborative):** run `claude`, start in plan mode, watch it read the failing test and related files (heavy exploration optionally pushed to an Explore subagent), review the plan, approve, switch to acceptEdits, watch the diff land, steer if it drifts. Eyes-on every consequential action, on your machine. Best for subtle tasks, sensitive codebases, or learning what the agent does.
- **Codex (cloud/async/delegate):** type the task into the web/desktop UI (or delegate from CLI/IDE), it spins up an isolated cloud container, runs tests behind an OS sandbox with network off, hands you a PR with cited logs minutes later. You never watched it; you review the diff — and queued four other tasks meanwhile. Best for well-scoped, parallelizable work where you'd rather review outcomes than steer process.
- **opencode (open/model-agnostic):** run `opencode` (reconnecting to a persistent session), route planning to a cheap model and the fix to a frontier model mid-session, work Plan→Build with per-tool permissions, stay in tmux over SSH. Best when you refuse lock-in, want per-task routing, need data sovereignty, or live on a remote box.

Tradeoffs sharpened: Claude Code trades parallelism and vendor-flexibility for inspectability, immediacy, and a deeply tuned single-vendor harness. Codex trades immediacy and steering for throughput, parallelism, and containment-based safety. opencode trades first-party model co-design and out-of-box polish for independence, per-task routing, and openness.

## Adjacent tools (brief, for contrast)
- **Cursor** — IDE-native reference. VS Code fork; **Composer** (fast multi-file edits), **Agent Mode** (plan-then-act), **Background Agents** (async, server-side, can pick up Linear tickets and open PRs). Cursor 2.x runs up to 8 parallel agents on isolated git worktrees, deepest multi-model selection in any editor. IDE-native + model-flexible corner.
- **Aider** — open-source, git-centric, terminal; the purest example of edit-format and context engineering *as* the product. **Repo-map** (tree-sitter AST parse ranked by a PageRank-style graph into a token-budgeted symbol index, default ~1K tokens) and **multiple edit formats** (editblock/search-replace, unified diff, whole-file, patch — matched to model capability). **Architect mode** splits a strong planner from a cheap editor. Auto-commits to git (git rollback is the safety net; no default sandbox). Model-agnostic via LiteLLM.
- **Windsurf** — the other major IDE agent (Cognition, post-acquisition), built around **Cascade**: automatic context (its "Awareness Engine" pulls files without @-mentions), more autonomous multi-step execution than Cursor's diff-by-diff model, plus Workflows (markdown recipes), cross-session Memories, and its own SWE-1.5 model. IDE-native + autonomous corner.

Placed on the axes: Cursor and Windsurf anchor the IDE-interface end (Windsurf more autonomous, Cursor more control + model-flexible); Aider anchors the minimalist open-source/git-native end where context engineering, not a big harness, is the differentiator.

## The convergent core beneath the divergent surfaces
Strip the packaging and the tools are strikingly the same: the identical fundamental loop (call model, execute tools, feed results back, repeat); the same tool/file/shell primitives (read, write/edit, glob, grep, bash); the same higher-level primitives by 2026 (subagents with isolated context, a plan/read-only mode, ask-the-user tools, parallel execution, sandboxing, memory files, MCP). And active consolidation on shared standards: **MCP** for tools, **AGENTS.md/CLAUDE.md** for project memory, both now stewarded (with Block's goose) by the Linux Foundation's **Agentic AI Foundation** (formed December 2025). Independent convergence of competing teams on near-identical harness skeletons is itself evidence there is a right shape and everyone is finding it.

What is *genuinely* different: execution locus (local vs cloud/async), model coupling (who owns the model and pricing), where the trust boundary sits (permission pipeline vs OS sandbox vs git rollback vs container), session persistence (fresh-per-session vs persistent daemon vs SQLite threads), and default autonomy (pair vs delegate). Almost everything else — tool list, loop, memory-file idea, subagent pattern, MCP — is shared substrate. As frontier models converge to within a point or two on benchmarks, the harness, not the model, is the differentiator, and among harnesses the differences that survive scrutiny are these few architectural bets, not the feature lists.

## Recommendations
1. **Start from execution locus and trust posture, not features.** Well-scoped/parallelizable and comfortable with code in an OS-sandboxed cloud container → Codex. Eyes-on steering on a trusted local machine → Claude Code or opencode. Refuse lock-in / need data sovereignty / remote box → opencode or Aider.
2. **Decide model coupling deliberately.** Most co-designed harness+model with accepted lock-in (and its policy risk — the 2026 crackdown) → Claude Code/Codex. Model-policy risk or per-task cost routing matters → model-agnostic (opencode, Aider).
3. **Match autonomy to blast radius.** Plan/read-only + deny-first on unfamiliar or production-critical repos regardless of tool; reserve high-autonomy/bypass for isolated containers or throwaway VMs where the environment is the safety boundary.
4. **Invest your own engineering in the 98.4%.** Harness decisions — AGENTS.md/CLAUDE.md discipline, permission/approval config, hooks for must-always-run guardrails, context-budget hygiene — determine outcomes more than model choice.
5. **Lean on converging standards to stay portable.** Author project memory in AGENTS.md (cross-tool), expose internal capability via MCP, keep tool config stable within sessions for cache efficiency.

Thresholds that change the recommendation: if async cloud agents reach parity with interactive steering on *subtle* (not just well-scoped) tasks, delegation becomes the default even for hard work; if a single model decisively pulls ahead on real-world SWE tasks, the model-agnostic insurance premium stops being worth the lost co-design; if OS-sandbox tooling becomes a frictionless default across all tools, the safety-model axis collapses as a differentiator.

## Caveats
- The 1.6%/98.4% figure is a cited community estimate on a leak-derived, partly-generated bundle — directional, not precise.
- The space moves weekly: versions, model names, star/MAU/revenue figures, pricing, feature availability are 2026 snapshots (several from vendor/tool-affiliated interviews). Claude Code's `auto` mode, agent teams, and 1,000-subagent workflows were research previews at writing.
- Much Claude Code internal detail derives from reverse-engineering (the MBZUAI paper, community analysis of a March 31 2026 sourcemap leak). Well-corroborated but not all officially documented; the loop codename "nO" is unconfirmed by Anthropic.
- Vendor/secondary framing skews promotional; treat superlatives ("most popular," "ahead of Claude Code") and MAU/revenue cautiously. The primary arXiv compaction-stage naming (budget reduction, snip, microcompact, context collapse, auto-compact) is authoritative over secondary sources.
- The Anthropic third-party-harness crackdown is contested: Anthropic frames it as ToS clarification and anti-abuse; critics (DHH, OpenClaw's Peter Steinberger) frame it as anticompetitive lock-in. Facts (dates, error messages) and interpretation both still playing out.
