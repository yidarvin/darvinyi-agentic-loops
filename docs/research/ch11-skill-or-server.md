# Skill or Server: A Decision Framework for Extending the Agent

Research reference for *Agentic Loops*, Chapter 11 (synthesis/decision chapter concluding the "Extending the Agent" arc). Current as of 2026. The reader has just finished deep treatments of both MCP (protocol, transports, primitives, building a server, security) and Skills (SKILL.md, progressive disclosure, code execution, building, security). This chapter's job is the rigorous comparison and the decision framework, not re-explaining either mechanism. Fast-moving area; version-pin and re-verify at build time, and note that several load-bearing facts are 2026-dated.

## TL;DR
- Use the "connection vs instruction" test: build an MCP server when the hard part is *access* (a live database, a SaaS API, auth to a third-party system, a stateful connection shared across many clients); build a Skill when the hard part is *judgment* (a workflow, a procedure, domain expertise, or deterministic bundled scripts the agent should run against tools it already has). They are complementary layers — MCP is connectivity, Skills are procedure — and mature agents use both.
- The economics are the crispest differentiator. A big MCP server injects tens of thousands of tokens of tool definitions upfront (GitHub's official server ~17,600 tokens; a 5-server setup ~55,000; Anthropic has measured 134,000 before optimization). Skills use progressive disclosure — ~100 tokens of metadata per skill until triggered, then <5k when loaded. As of January 2026, MCP Tool Search narrows much of that gap (~85% reduction, ~77k→~8.7k for 50 tools), but the architectural split remains.
- The default answer for real production work is "both, layered." Skills should not try to open network connections or hold credentials (no built-in auth model, ship as static files); MCP servers should not be built to encode static team knowledge (over-engineering). The winning pattern is a Skill that orchestrates MCP tools: the Skill supplies know-how, the MCP server supplies connectivity.

## The fundamental architectural distinction
**MCP is connectivity.** A client-server protocol; a server is a running process (local stdio or remote Streamable HTTP) the agent talks to over JSON-RPC, exposing tools/resources/prompts. There is a network/process boundary and a live, potentially stateful connection to something external. Mahesh Murag: MCP servers are "tools and data from the outside world," and MCP "fits in as basically that entire bottom layer" of the agent architecture, letting agents "expand after they've been initialized to start discovering different capabilities."

**A Skill is instruction.** A folder (minimum: one `SKILL.md` with YAML frontmatter + markdown body) that lives in the agent's own execution environment and loads via progressive disclosure. No wire, no server, no protocol. Zhang and Murag described the converged architecture as "an agent loop connected to the file system, MCP servers, and a library of Skills."

Crispest one-liners in the discourse: Block's Goose team — "Skills describe the workflow. MCP provides the runner." Damian Galarza — "MCPs are verbs. Skills are playbooks." Sumo Logic's test — "If a senior expert on your team was doing this manually, would the hard part be getting access to the data? Or knowing what to do with it once they had it? If the hard part is access, you need MCP. If the hard part is judgment, you need a skill."

**What each is NOT.** MCP does not give procedural knowledge about *how* to use a tool well in your context — it grants access. A Skill does not, on its own, open a socket, authenticate to Salesforce, or hold a connection pool — it's a static artifact the agent reads. Armin Ronacher's mechanical subtlety: a skill "does not actually load a tool definition into the context. The tools remain the same: bash and the other tools the agent already has. All it learns from the skill are tips and tricks for how to use these tools more effectively."

## Dimension-by-dimension comparison
**Context / token economy (the sharpest axis).** MCP loads tool definitions upfront, every request: GitHub official server ~17,600 tokens (94 tools; fuller toolsets run higher, one analysis cites 55,000 for 93 tools); Atlassian ~10,000; a typical 5-server / ~58-tool setup ~55,000 before the first character; Anthropic's worst case "134K tokens before optimization"; per-tool ~200–1,400 tokens. Skills invert it: metadata ~100 tokens/skill, full instructions <5k only when triggered, bundled files only as needed. Anthropic's cookbook Excel case: "~8,000 tokens to explain all Excel features upfront" vs "~5,000 tokens only when Excel skill is invoked" (~98% initial savings). Practitioner comparison: 20 skills ~1,000 tokens vs 18,000+ if bodies always loaded.

*The nuance — MCP's token problem is being fixed.* (1) **MCP Tool Search** (Anthropic, Jan 14 2026): when definitions exceed 10% of context, tools are `defer_loading: true` and discovered on-demand; ~85% reduction (~77k→~8.7k for 50 tools); accuracy gains too (Opus 4 49%→74%, Opus 4.5 79.5%→88.1%). Willison: "context pollution is why I rarely used MCP, now that it's solved there's no reason not to hook up dozens or even hundreds of MCPs." (2) **Code execution with MCP** (presenting servers as code APIs called from a sandbox): up to 98.7% overhead reduction. (3) **Tool filtering / gateways** (allowedTools lists; gateways reporting 58–92% reductions). (4) **Prompt caching** amortizes the *billing* cost across turns but not the *attention/working-memory* cost (per modelcontextprotocol issue #2808: "10,000 tokens of schemas occupy attention slots"). Honest framing: Skills were lazy-loaded from day one; MCP's lazy loading is a 2026 retrofit that largely closes the *startup* gap but leaves the architectural split intact.

**Where code runs.** MCP server code runs in a separate process (local subprocess over stdio, or remote web service). Skill code runs in the agent's own sandbox; `scripts/` are executed by the agent, not loaded into context. Skills' biggest dependency: they require a code-execution environment. Willison: "the biggest difference... a significant dependency, but it's somewhat bewildering how much new capability it unlocks."

**State and connectivity.** MCP maintains a live stateful connection (pools, transactions, session state) — why dedicated servers exist for Postgres, MongoDB, Snowflake. Skills are stateless folders; scripts *can* call the network if the environment permits, but on the managed API skills "cannot make external network calls or install runtime packages."

**Authentication and remote access.** MCP has a formal framework: March 2025 made **OAuth 2.1** (Auth Code + PKCE) standard for remote HTTP servers; November 2025 banned plain PKCE (S256 only), required RFC 8707 resource params, mandated `WWW-Authenticate` on 401s. Remote servers act as OAuth resource servers validating short-lived tokens. Skills have **no built-in auth model** — local files; any credentials come from the environment. Decisive: if you need OAuth to a third-party service, that's an MCP concern (or the service's hosted MCP server), not a Skill.

**Real-time / live data.** MCP pulls live data by design. A Skill's bundled reference files are static — the ByteByteGo rule: "if the data changes between invocations, you need MCP... If the knowledge is stable enough that you could write it down once and have it be right for weeks, a skill file is simpler."

**Distribution and portability.** MCP servers publish to package registries (npm/PyPI/Docker) with metadata in the official MCP Registry (reverse-DNS namespaces), or hosted as remote URLs; require installation + a running process. Skills: a folder/zip copy-pasted into `.claude/skills/`. The same `SKILL.md` works unmodified across Claude Code, Codex CLI, Gemini CLI, Copilot, Cursor, VS Code. (A script-heavy skill assumes the target runtime has the right deps — the spec doesn't enforce that, so it "may break silently on a different machine.")

**Reusability across surfaces.** MCP's portability is *developer-side and cross-client* (build one server, any client connects — "a shared developer incentive"). Skills' portability is *artifact-level* (the file travels).

**Determinism and reliability.** A Skill can bundle deterministic scripts (e.g., document skills' OOXML validation/pack). MCP tools are deterministic server-side. Key difference (Phil Whittaker): with MCP "when you trigger a tool, you know you're triggering it," with SDKs/testing/mature infra; with Skills, *triggering* is probabilistic (model decides from the description). LlamaIndex: MCP gives "predictable and precise execution."

**Development complexity.** Skill: write a markdown file, optionally add scripts (skill-creator scaffolds it). MCP server: implement protocol + transport + server code + deployment + (remote) OAuth 2.1. Willison called the Skills spec "deliciously tiny"; MCP "a full protocol specification... requires implementing a server."

**Security model.** MCP centralizes risk at a server trust boundary and adds protocol-level vectors (tool poisoning, injection via tool results); a compromised shared server affects every connected agent (exemplar: **CVE-2025-6514**, mcp-remote, CVSS 9.6, JFrog, "the first time that full remote code execution is achieved... connecting to an untrusted remote MCP server"; 437,000+ downloads). Skills distribute risk as arbitrary-code supply chain (Snyk "ToxicSkills," Feb 5 2026, 3,984 skills from ClawHub/skills.sh: "36.82% have at least one security flaw," 13.4% critical, 76 malicious payloads; Liu et al. "Agent Skills in the Wild," arXiv 2601.10338, 42,447 skills: 26.1% with ≥1 vulnerability). Counter-asymmetry (Noma): "Skills resist rug-pull attacks because they are usually static files requiring manual updates, whereas MCP servers pinned to @latest fetch new package versions on every agent load" — but Datadog counters that skill dynamic context (`!` commands) can execute before the model sees the content.

**Maintenance and operations.** A Skill is a static artifact (version like a runbook, no hosting). An MCP server is live infrastructure (hosting, uptime, auth rotation, versioning). Governance cuts both ways: a server is a central chokepoint you can observe/audit/rate-limit ("One MCP config, consumed by many — auditable and swappable"); skills "run locally... invisible to central systems" (hence enterprise demand for central skill management; Anthropic shipped org-wide admin provisioning Dec 18 2025).

## The decision framework
**Single question:** is the hard part access or judgment? Access → MCP. Judgment → Skill. Both → both.

**Build a Skill when:** you need to encode a workflow/procedure/domain expertise/best practices; you want deterministic bundled scripts run against tools the agent already has; the capability is about *how*, not *what to reach*; you want maximum simplicity/portability/token efficiency; you don't need live connectivity, auth, or shared-server governance.

**Build an MCP server when:** you must connect to an external system (database, SaaS API, live data); you need auth to a third-party service; you need a stateful connection; you're exposing a shared capability to many agents/clients over a network with central governance; the capability requires a running backend.

**Use BOTH when** a skill describes the workflow and an MCP server provides the tools it references — the production default. (Incident triage: the Skill defines process + hypotheses; MCP tools like `list_recent_deploys` execute each step.)

**Decision tree (synthesis):**
1. Requires reaching a live external system, holding state, or authenticating to a third party? → **Yes: MCP** (or existing server). No → continue.
2. Can the agent already do the underlying action (write code, run a CLI, manipulate files) and you mainly need to teach it *how*? → **Yes: Skill.**
3. Must the capability be shared across many agents/clients with central governance? → **Yes: MCP** (add a Skill on top if judgment also needed).
4. Need both access *and* judgment? → **Both: MCP for connectivity + a Skill that orchestrates it.**
5. Before building a server, ask: does a CLI or existing tool already do this? If the agent can shell out, a **Skill wrapping the CLI** often beats a bespoke server.

## Concrete worked examples
- **(a) Internal Postgres access → MCP server.** Hard part is access/state/pooling; canonical MCP; dedicated Postgres servers exist for persistent transactional connections.
- **(b) Brand guidelines + report formatting → Skill.** Pure judgment/consistency, no external system.
- **(c) Connect to GitHub → MCP server (existing one).** External SaaS, auth, live data. Caveat: the full server is heavy — use tool filtering / Tool Search or a scoped read-only server.
- **(d) Fill our specific PDF forms correctly → Skill with bundled scripts.** Anthropic's flagship example: Claude understands PDFs but "is limited in its ability to manipulate them directly"; the skill adds it via bundled scripts + reference files (`forms.md`), not a server.
- **(e) Live Slack access + team communication norms → both.** MCP (or Slack connector) for connection + auth; a Skill for tone, escalation rules, when-to-post norms.
- **(f) Document creation (docx/pptx/xlsx) → Skill.** Bundles workflows + deterministic scripts (OOXML unpack/validate/pack, HTML→PPTX via headless browser, tracked-changes redlining). No server; on the API invoked as managed skills alongside `code_execution`.

Negative example (dibi8): "analyze our metrics" as a beautiful skill when the agent *can't reach the metrics* — "No instructions substitute for the missing MCP connection." Judgment without access is an expensive search engine; access without judgment is "a faster way to make the same mistakes."

## How they compose
Layered agent architecture, bottom to top: agent loop → runtime (filesystem + code execution) → MCP servers (connectivity) → Skills (procedure). A Skill orchestrates MCP tools by referencing them by name.

Real patterns: **Sentry** — the MCP server provides connectivity (query issues/traces, `use_sentry`), and Sentry *also* ships skills (`sentry-fix-issues`, `sentry-code-review`) encoding the debugging workflow on top ("MCP gave us an open foundation... With skills we can turn that context into focused workflows"). Note the real friction Ronacher documents: he moved *off* the Sentry MCP to a hand-maintained skill because the MCP cost ~8k tokens on load and its tool descriptions changed without notice — a caution that skill-over-MCP wrappers are brittle when the underlying server lacks API stability. Other patterns: a "Deploy & Notify" skill using GitHub + CI/CD + Slack MCPs; a database-workflow skill wrapping DB MCP tools (MCP connects, the skill defines which analyses and how to interpret, scripts do the stats).

Why "connection + instruction" is complete: the layers evolve independently — a domain expert improves a skill without touching servers; IT swaps a data provider by changing an MCP URL. Anthropic deployed exactly this in financial-services and life-sciences verticals ("each came with a set of MCP servers and a set of skills").

Live counter-current (flag honestly): **Code Mode** (Cloudflare, widely adopted) — expose a sandbox where the agent writes TypeScript to orchestrate tools ("LLMs are better at writing code to call tools than at calling tools directly"); Ronacher's stronger position — "moving all my MCPs to skills," preferring the agent to write/maintain CLI-wrapping tools. These blur but don't erase the boundary — MCP still typically sits underneath as the adapter to real services with auth and state.

## Industry debate
**"Will Skills replace MCP?"** — sparked by Willison's Oct 2025 "Claude Skills are awesome, maybe a bigger deal than MCP," predicting a "Cambrian explosion" and citing MCP token cost for his "waning interest" ("you don't need MCP for prompt templates and context resources — those are skills").

**Consensus rebuttal: complementary.** Anthropic (Murag): "MCP provides secure connectivity to external software and data, while skills provide the procedural knowledge for using those tools effectively." Phil Whittaker: "I very much disagree that they're a replacement for MCP." Umbraco keeps its 330-tool MCP strategy unchanged. Speakeasy calls "Skills vs MCP" a "false dichotomy," analogizing to "documentation vs APIs — obviously you need both."

**Genuine disagreement / uncertainty:** *Do many MCP servers deserve to exist?* Ronacher: "many MCP servers don't need to exist. They're either bad API wrappers or they're actually truly replaceable by a skill." Maxim: Skills "supplanted many naive MCP servers that never needed to exist... Skills didn't replace MCP broadly." *Can skills+sandbox subsume MCP?* Speakeasy's rebuttal: MCP still owns persistence, pooling, hard OAuth (Salesforce launched hosted MCP servers precisely because its OAuth has "enough quirks"). *Latency:* fresh interpreter per skill-script call adds ~2s; a resident server loads once (compounds in MCP's favor for long pipelines). *Determinism of triggering:* skills' description-based triggering is probabilistic and can misfire; MCP calls are explicit.

**Hype vs reality:** "Skills killed MCP" is overstated — both adoption curves rose in parallel through early 2026. Per Anthropic (Dec 9 2025), MCP had "over 97 million monthly SDK downloads, 10,000 active servers and first-class client support across major AI platforms." Measured take (Maxim): "Skills did replace MCP for some use cases — specifically, the ones that were over-engineered to begin with. But MCP didn't die. It found its proper scope." Both went to open governance: MCP donated to the Agentic AI Foundation (Dec 9 2025, "a directed fund under the Linux Foundation, co-founded by Anthropic, Block and OpenAI, with support from Google, Microsoft, AWS, Cloudflare and Bloomberg"); Agent Skills published as an open standard at agentskills.io (Dec 18 2025).

## Anti-patterns and guidance
**Anti-pattern 1 — MCP-server-for-everything (over-engineering).** Shipping a service to solve a docs/workflow problem. "Building an MCP server for GitHub, AWS, or Docker commands was always overengineered" — the agent already handles `gh pr create`; a Skill documenting conventions is simpler. If you're not connecting to an external system, you probably don't need a server.
**Anti-pattern 2 — Cramming connectivity into a Skill.** Writing "this process uses the DeepL API" as a skill when you need `deepl-mcp`. Real-time external data, auth, or persistent connection → MCP.
**Anti-pattern 3 — Not using them together.** A gorgeous skill for a task the agent can't reach, or a rich server with no procedural guidance ("you gave the agent the keys; you didn't teach it to drive").
**Anti-pattern 4 — The bloated CLAUDE.md.** Stuffing every procedure into always-on context; move situational playbooks into skills.

**Team guidance:** start with the simplest thing, layer up (repeated multi-step sequence → write a Skill); prefer official/existing servers; prefer a CLI-wrapping Skill over a bespoke server when the agent can shell out; keep servers to ~3–5, use Tool Search / filtering, scope servers narrowly (read-only variants); treat skills as first-class software (version, test, evaluate, review); govern both (central MCP config; central skill provisioning for Team/Enterprise).

**Security (both):** MCP — validate tokens (signature, expiry, audience, scope), don't trust connect-time tool descriptions blindly, treat tool results as untrusted input. Skills — audit before installing, avoid blind `@latest`, restrict shell execution, review `.claude/` changes, collect telemetry. System-level: least-privilege tools, human-in-the-loop for sensitive actions, planner/executor split.

**Migration:** *Server → Skill* if a server is a thin wrapper encoding static knowledge (watch the API-stability trap on hand-maintained summaries). *Skill → Server* if scripts keep needing live data/auth/shared state. *Toward "both":* most setups converge on Skill-orchestrates-MCP; "almost nobody designs for it from the start" but should.

**Future outlook:** the boundary keeps blurring — MCP Tool Search brings skill-like progressive disclosure to tools; MCP Apps (Jan 2026) add interactive UI; a mid-2026 spec revision targets stateless-yet-stateful operation; early work serves `SKILL.md` over MCP (`skill://` resources). Anthropic's ambition: agents that "create, edit, and evaluate Skills on their own." A third layer is emerging above both — Agent-as-a-Service (Manus, Devin, Claude Managed Agents beta): a Skill sells a procedure, an MCP server sells a capability, an AaaS endpoint sells a result. Durable takeaway: the connection-vs-instruction split is architectural and will outlast the token-economics arguments.

## Recommendations (staged)
1. **Classify the capability first, every time:** is the hard part access or judgment? Write the answer as a sentence. If you can't decide, it's probably both.
2. **Default to the cheapest sufficient mechanism:** judgment-only → Skill (markdown file; scripts only if you need determinism); access-only + existing server → adopt it, apply tool filtering / Tool Search; access-only, no server, agent can shell out → Skill wrapping the CLI before building a server; access needing a bespoke backend/auth/shared governance → build the MCP server (OAuth 2.1 if remote).
3. **Layer for production:** once used in real workflows, add the missing layer — judgment in a Skill on top of the server, or promote recurring connectivity out of a skill into a server. Target Skill-orchestrates-MCP.
4. **Govern and secure:** route MCP through a central auditable config/gateway; provision skills centrally; treat both as reviewed, versioned software; least-privilege + human-in-the-loop on state-changing actions.

**Thresholds that change the decision:** MCP tool definitions >~10% of context (Anthropic's Tool Search trigger) or past ~30–50 tools (selection accuracy degrades) → enable Tool Search / filtering or split servers (don't reach for Skills to fix a connectivity need). Past ~3–5 concurrent servers → consolidate via a gateway. Knowledge stable "for weeks" → Skill; changes between invocations → MCP. More than one agent/surface needs the same capability with governance → favor a server. Pipelines with hundreds of tool calls → prefer a resident server over per-call interpreter starts.

## Caveats
- Fast-moving ground truth: MCP Tool Search (Jan 2026), the Nov 2025 OAuth revision, MCP Apps (Jan 2026), and a planned mid-2026 revision are all recent/evolving. Token figures for the same server vary widely by method, tool subset, and date — treat all specific numbers as order-of-magnitude.
- Marketing/vendor spin: many token-reduction percentages (58–92%, "98.7%," "95%") come from vendors selling the fix — direction reliable, magnitude not independently audited. Anthropic's own figures (85% reduction; 49%→74% accuracy) are first-party best-case internal evals.
- Security statistics are early and heterogeneous: the skill-vulnerability audits (Snyk 36.82% of 3,984; Liu et al. 26.1% of 42,447) come from different corpora and *third-party* marketplaces (notably OpenClaw/ClawHub — Koi Security's Feb 2026 audit of all 2,857 ClawHub skills flagged 341 malicious, 335 from one "ClawHavoc" campaign), *not* Anthropic's curated first-party directory; don't read them as the risk profile of vetted skills.
- Some datapoints are secondary (ecosystem growth rates, per-server token counts) from aggregators/blogs; they establish scale, not precision. Anthropic's MCP numbers (97M monthly downloads, 10,000 servers) are first-party as of Dec 9 2025.
- The debate is unsettled at the margins: whether skills+sandbox will absorb simple MCP servers, and how much bespoke-server-building is warranted, remain live disagreements among credible practitioners (Ronacher vs the Anthropic/Speakeasy "both" camp). This chapter takes "complementary, layer both" as the decision-useful default while flagging the dissent.
